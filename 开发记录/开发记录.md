## 第01章：项目初始化与基础配置 | 2026-07-01

### 概述

创建 Spring Boot 3 项目骨架，配置 Git 远程仓库，搭建通用基础代码（统一响应体、异常处理、跨域、分页等）。

### 一、Git 远程仓库关联

1. 添加远程仓库 origin: `https://github.com/vioabio/AI-Agent.git`
2. 初始提交 12 个文件到本地仓库（commit `60f7dcc`）
3. 强制推送到远程 `main` 分支
4. 本地 `master` 分支重命名为 `main`，设置上游跟踪 `origin/main`

**Git 配置要点**：
- 项目层面设置 `core.safecrlf=false` 解决 `mvnw.cmd` 无法添加问题
- `core.autocrlf=false` 防止 Windows 下 CRLF 自动转换

### 二、统一响应包装类

#### `common/Result<T>` — 统一 API 响应体

- `code` / `message` / `data` / `timestamp` 四个字段
- 静态工厂方法：`success()`、`error()`
- 链式构造，兼容 `ResultCode` 枚举和自定义消息
- `@JsonInclude(NON_NULL)` 自动忽略空值

#### `common/ResultCode` — 响应状态码枚举

| 分类 | 范围 | 示例 |
|------|------|------|
| HTTP 标准 | 200~503 | SUCCESS、BAD_REQUEST、UNAUTHORIZED、NOT_FOUND、INTERNAL_ERROR |
| 业务异常 | 1xxx | BUSINESS_ERROR(1000)、PARAM_ERROR(1001)、DATA_NOT_FOUND(1002) |
| 系统异常 | 2xxx | SYSTEM_ERROR(2000)、RPC_ERROR(2001)、DB_ERROR(2002) |

### 三、自定义异常

#### `exception/BusinessException` — 业务异常基类

- 继承 `RuntimeException`，携带 `code` 和 `message`
- 支持多种构造方式：自定义消息、`ResultCode`、带异常链
- 统一在 `GlobalExceptionHandler` 中捕获处理

### 四、全局异常处理器

#### `exception/GlobalExceptionHandler`

`@RestControllerAdvice` 全局拦截异常，覆盖 11 种异常类型 + 兜底：

| 异常类型 | 处理方式 |
|----------|----------|
| `BusinessException` | 返回对应业务状态码和消息 |
| `MethodArgumentNotValidException` | 提取字段校验错误并拼接 |
| `BindException` | 提取表单绑定错误 |
| `ConstraintViolationException` | 提取约束校验错误 |
| `MissingServletRequestParameterException` | 提示缺少必要参数 |
| `MethodArgumentTypeMismatchException` | 提示参数类型错误 |
| `HttpMessageNotReadableException` | 提示请求体格式错误 |
| `HttpRequestMethodNotSupportedException` | 提示请求方法不支持 |
| `MaxUploadSizeExceededException` | 提示上传文件过大 |
| `NoHandlerFoundException` / `NoResourceFoundException` | 404 |
| `Exception` (兜底) | 500，日志记录完整堆栈 |

### 三、提示词工程学习

#### 基础技巧

| 技巧 | 说明 | 核心要点 |
|------|------|----------|
| 明确指定任务和角色 | 为 AI 提供清晰的任务描述和角色定位 | 帮助模型理解背景和期望 |
| 提供详细说明和具体示例 | 提供足够的上下文信息和期望的输出格式 | 减少模型的不确定性 |
| 使用结构化格式引导思维 | 通过列表、表格等结构化格式 | 使指令更易理解，输出更有条理 |
| 明确输出格式要求 | 指定输出的格式、长度、风格等要求 | 获得更符合预期的结果 |

#### 进阶技巧

| 技巧 | 说明 | 适用场景 |
|------|------|----------|
| 思维链提示法（Chain-of-Thought） | 引导模型展示推理过程，逐步思考 | 提高复杂问题的准确性 |
| 少样本学习（Few-Shot Learning） | 提供几个输入-输出对示例 | 帮助模型理解任务模式 |
| 分步骤指导（Step-by-Step） | 将复杂任务分解为可管理的步骤 | 确保模型完成每个关键环节 |
| 自我评估和修正 | 让模型评估自己的输出并改进 | 提高准确性和质量 |
| 知识检索和引用 | 引导模型检索相关信息并明确引用来源 | 提高可靠性 |
| 多视角分析 | 从不同角度、立场或专业视角分析 | 提供全面见解 |

**核心原则**：任务越复杂，就越要给 Prompt 补充更多细节。把 AI 当成人类，模糊的问题得到模糊的答案。

## 第03章：AI 恋爱大师应用 | 2026-07-02

### 概述

基于 Spring AI 框架开发「AI 恋爱大师」应用，涵盖 ChatClient、Advisor、ChatMemory、结构化输出、RAG 知识库基础、SSE 流式对话等 Spring AI 核心特性。

**参考源码**：`D:\计算机项目\鱼皮\yu-ai-agent-master`

### 一、开发流程

#### 1. 多轮对话 AI 应用 — `app/LoveApp.java`

使用 Spring AI ChatClient + MessageChatMemoryAdvisor 实现：

```java
@Component
public class LoveApp {
    private final ChatClient chatClient;
    private static final String SYSTEM_PROMPT = "扮演深耕恋爱心理领域的专家...";

    public LoveApp(ChatModel dashscopeChatModel) {
        MessageWindowChatMemory chatMemory = MessageWindowChatMemory.builder()
                .chatMemoryRepository(new InMemoryChatMemoryRepository())
                .maxMessages(20).build();
        chatClient = ChatClient.builder(dashscopeChatModel)
                .defaultSystem(SYSTEM_PROMPT)
                .defaultAdvisors(
                    MessageChatMemoryAdvisor.builder(chatMemory).build(),
                    new MyLoggerAdvisor()
                ).build();
    }
}
```

核心方法：

| 方法 | 说明 |
|------|------|
| `doChat(message, chatId)` | 同步对话，返回完整响应 |
| `doChatByStream(message, chatId)` | 流式对话，返回 `Flux<String>` 支持 SSE |
| `doChatWithReport(message, chatId)` | 结构化输出，返回 `LoveReport` 对象（含 title + suggestions） |

#### 2. 自定义 Advisor

| 文件 | 功能 | 说明 |
|------|------|------|
| `MyLoggerAdvisor.java` | 日志拦截器 | 同时实现 `CallAroundAdvisor` 和 `StreamAroundAdvisor`，记录每次请求和响应 |
| `ReReadingAdvisor.java` | Re2 推理增强 | 让模型重新阅读问题以提高推理能力（成本加倍，C端不建议，已注释） |

**Advisor 执行流程**：前置增强（改写 Prompt）→ 调用 ChatModel → 后置增强（记录日志/处理结果）

#### 3. 结构化输出 — 恋爱报告功能

定义 `LoveReport` record（title + suggestions），Spring AI 自动将 Java 类转为 JSON Schema 附加到提示词中，AI 生成符合格式的 JSON → 自动反序列化为 Java 对象。

#### 4. 对话记忆持久化 — `chatmemory/FileBasedChatMemory.java`

- 实现 `ChatMemory` 接口，使用 **Kryo** 序列化库将对话持久化到磁盘
- 存储位置：`${user.dir}/tmp/chat-memory/`
- 解决了 `Message` 接口多子类、无 Serializable 实现的问题
- 当前 LoveApp 使用内存存储，文件持久化可按需切换

#### 5. REST API 控制器 — `controller/AiController.java`

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/ai/love_app/chat/sync` | GET | 同步调用 |
| `/api/ai/love_app/chat/sse` | GET | SSE 流式（`Flux<String>`） |
| `/api/ai/love_app/chat/server_sent_event` | GET | ServerSentEvent 流式 |
| `/api/ai/love_app/chat/sse_emitter` | GET | SseEmitter 流式（3分钟超时） |

#### 6. RAG 知识库基础文档 — `resources/document/`

### 四、技术要点总结

1. **ChatClient 优于 ChatModel**：支持链式调用、默认系统提示词、Advisor 拦截器、结构化输出
2. **Advisor 即拦截器**：责任链模式，前置/后置增强，可自定义日志、鉴权、Prompt 改写
3. **MessageChatMemoryAdvisor 是首选**：保留原始对话结构，更符合现代 LLM 对话模型设计
4. **结构化输出原理**：调用前附加 JSON Schema，调用后将 JSON 转为 Java 对象
5. **Kryo 解决序列化难题**：Message 接口多子类、无 Serializable，Kryo 是合适方案
6. **PromptTemplate 实现前后端分离**：提示词从代码抽离到文件，便于版本管理
7. **MVP 策略**：先核心对话，再逐步扩展知识库、工具调用、MCP 等高级能力
8. **Token 成本意识**：系统提示词、用户提示词和 AI 输出都消耗成本，需持续优化

### 二、完善 ETL 流程

#### `LoveAppVectorStoreConfig.java` — 完整 ETL 流水线

```java
// 1. ETL — 加载文档
List<Document> documentList = loveAppDocumentLoader.loadMarkdowns();
// 2. ETL — 文档切割（TokenTextSplitter，chunkSize=200, overlap=100）
List<Document> splitDocuments = myTokenTextSplitter.splitCustomized(documentList);
// 3. ETL — 元信息增强（KeywordMetadataEnricher，AI 提取 5 个关键词）
List<Document> enrichedDocuments = myKeywordEnricher.enrichDocuments(splitDocuments);
// 4. ETL — 写入向量数据库（SimpleVectorStore 内存存储）
simpleVectorStore.add(enrichedDocuments);
```

#### ETL 三大组件

| 组件 | 实现 | 说明 |
|------|------|------|
| **DocumentReader（抽取）** | `LoveAppDocumentLoader` | 从 classpath:document 加载 Markdown 文档，提取 status 元数据 |
| **DocumentTransformer（转换）** | `MyTokenTextSplitter` + `MyKeywordEnricher` | Token 级别文档切割 + AI 自动关键词标注 |
| **DocumentWriter（写入）** | `SimpleVectorStore` / `PgVectorStore` | 本地内存存储 / 生产级 PostgreSQL 向量存储 |

### 三、PGVector 生产级向量存储

#### `PgVectorVectorStoreConfig.java`

- `@Configuration` 默认注释，避免无 PostgreSQL 环境启动报错
- 补充完整 ETL 流程（加载 → 切割 → 增强 → 写入）

| 参数 | 值 | 说明 |
|------|-----|------|
| dimensions | 1536 | 向量维度 |
| distanceType | COSINE_DISTANCE | 余弦相似度 |
| indexType | HNSW | 层次可导航小世界图索引 |
| initializeSchema | true | 自动建表 |
| vectorTableName | vector_store | 向量存储表名 |
| maxDocumentBatchSize | 10000 | 最大批处理文档数 |

### 四、上下文查询增强器

#### `LoveAppContextualQueryAugmenterFactory.java`

当检索不到相关文档时，返回友好提示而非让 AI 胡编乱造：

### 二、MCP 调用本质

MCP 调用的本质是**工具调用的标准化**：

1. MCP Server 启动后，通过 MCP 协议向 Client 宣告自己提供了哪些工具
2. 主应用的 `ToolCallbackProvider` 从所有 MCP Server 收集工具列表
3. 用户发送消息时，AI 看到本地工具 + MCP 工具的全部能力
4. AI 决定调用某个 MCP 工具 → 主应用通过 MCP 协议向 MCP Server 发起调用 → MCP Server 执行并返回结果 → AI 总结回复

**不是 AI 服务器直接调用 MCP 服务**，而是 AI 告诉后端程序"我想调用哪个工具"，后端程序代为执行。

## 第08章：AI 智能体构建 | 2026-07-04

### 概述

基于 Spring AI 框架自主实现具备 ReAct（Reasoning + Acting）模式的 AI 超级智能体 VioManus。采用模板方法设计模式，构建 BaseAgent → ReActAgent → ToolCallAgent → VioManus 四层继承体系，使智能体拥有自主规划、工具调用、多步推理和主动终止的能力。

**参考源码**：`D:\计算机项目\鱼皮\yu-ai-agent-master`

### 一、开发流程

#### 步骤 1：创建 `AgentState` 状态枚举

```java
public enum AgentState {
    IDLE,       // 空闲 — Agent 尚未开始或已执行完毕
    RUNNING,    // 运行中 — Agent 正在执行 Agent Loop
    FINISHED,   // 已完成 — 任务正常完成
    ERROR       // 错误 — 执行过程中发生异常
}
```

状态转换：`IDLE → RUNNING → FINISHED` 或 `IDLE → RUNNING → ERROR`。从非 IDLE 状态再次调用 `run()` 会抛出异常。

#### 步骤 2：创建 `BaseAgent` 抽象基类

智能体框架的核心基类：

| 属性 | 类型 | 说明 |
|------|------|------|
| `name` | String | 智能体名称 |
| `systemPrompt` | String | 系统提示词，定义角色和能力边界 |
| `nextStepPrompt` | String | 每步执行前追加的引导提示词 |
| `state` | AgentState | 当前状态（初始 IDLE） |
| `currentStep` | int | 当前步数（从 1 开始） |
| `maxSteps` | int | 最大步数限制（默认 10，安全阀） |
| `chatClient` | ChatClient | AI 对话客户端（由子类注入，不写死大模型） |
| `messageList` | List\<Message\> | 消息上下文列表（自主维护） |
| `stopped` | volatile boolean | 手动停止标志 |

**两个核心执行方法**：
- **`run(String userPrompt)`** — 同步执行，返回所有步骤结果的拼接字符串
- **`runStream(String userPrompt)`** — SSE 流式执行，通过 `CompletableFuture.runAsync()` 异步运行，每步结果实时推送到 `SseEmitter`（5 分钟超时）

**Agent Loop 核心逻辑**：

```
1. 状态校验（必须是 IDLE）
2. state = RUNNING，记录用户消息
3. for (i = 0; i < maxSteps && state != FINISHED && !stopped; i++)
      step() → 收集结果
4. 超步数检测 → state = FINISHED
5. 异常捕获 → state = ERROR
6. finally: cleanup() 清理资源
```

#### 步骤 3：创建 `ReActAgent` 抽象类

实现 ReAct（Reasoning + Acting）模式：

```java
@Override
public String step() {
    boolean shouldAct = think();   // 思考：需要行动吗？
    if (!shouldAct) return "思考完成 - 无需行动";
    return act();                   // 行动：执行工具调用
}
```

#### 步骤 4：创建 `ToolCallAgent` — 最核心的类

具体实现了 `think()` 和 `act()` 方法。

### 五、技术要点总结

1. **模板方法模式是 Agent 框架的核心**：BaseAgent 定义 Agent Loop 骨架，子类通过 `step()`/`think()`/`act()` 填充具体行为
2. **`withInternalToolExecutionEnabled(false)` 是 Agent 与普通 ChatClient 的本质区别**：Agent 需要在循环中多次调用 AI + 多次执行工具，必须自主管理消息上下文
3. **`doTerminate` 工具是 Agent 自主终止的唯一正常途径**：没有它，Agent 只能被动等待 maxSteps 耗尽
4. **每次请求必须创建新的 Agent 实例**：Agent 维护内部状态，跨请求共享会导致状态冲突
5. **`nextStepPrompt` 是引导 AI 主动规划的关键**：每步 Think 前都会追加此提示词
6. **`maxSteps` 是成本和安全的平衡点**：步数太少复杂任务无法完成，步数太多 Token 浪费严重
7. **Agent 没有长期记忆**：简单多轮对话应使用 LoveApp，复杂自主任务才用 VioManus
8. **ReAct 模式的本质是"思考-行动-观察"循环**：每次工具执行的结果自动成为下一轮思考的上下文

```js
  const eventSource = new EventSource(fullUrl)
```

#### 合并逻辑

```java
private ToolCallback[] getMergedTools() {
    if (toolCallbackProvider == null) return availableTools;
    ToolCallback[] mcpTools = toolCallbackProvider.getToolCallbacks();
    if (mcpTools == null || mcpTools.length == 0) return availableTools;
    // 合并本地工具 + MCP 远程工具
    ToolCallback[] merged = Arrays.copyOf(availableTools,
            availableTools.length + mcpTools.length);
    System.arraycopy(mcpTools, 0, merged, availableTools.length, mcpTools.length);
    return merged;
}
```

## 补充开发：Spring AI 版本冲突修复 | 2026-07-06

### 一、`ClassNotFoundException: FunctionCallbackResolver`

#### 根因

`spring-ai-alibaba-starter` 1.0.0-M6.1（旧一体式 Milestone）绕过 BOM 管理，其内部的 `DashScopeAutoConfiguration` 引用 `FunctionCallbackResolver` 类，该类在 Spring AI 1.0.0 GA 中已被移除。

#### 修复

将旧一体式 M6.1 starter 替换为新版模块化 dashscope starter（版本由 BOM 管理）：

```diff
- <artifactId>spring-ai-alibaba-starter</artifactId>
- <version>1.0.0-M6.1</version>
+ <artifactId>spring-ai-alibaba-starter-dashscope</artifactId>
```

全部依赖统一为 GA 版本，项目编译零错误，无需修改 Java 源码。

### 二、VioManus `@Component` 导致启动失败

#### 根因

`VioManus` 标注了 `@Component`，Spring 尝试创建单例 Bean 时调用无参构造器失败。深层原因：VioManus 是**有状态**的，设计上不能作为单例跨请求共享，每次请求在 Controller 中手动 `new` 创建新实例。

#### 修复

删除 `VioManus` 上的 `@Component` 注解。

## 补充开发：OpenAPI 接口文档修复 + Swagger 注解规范化 | 2026-07-08

### 概述

修复 `springdoc-openapi` + `openapi-typescript-codegen` 生成 TypeScript 前端代码的 4 个 bug，建立「后端 OpenAPI → 前端类型代码」自动生成流水线。

### 问题与修复

| # | 问题 | 根因 | 修复 |
|---|------|------|------|
| 1 | 所有端点 500 响应含 `ResultVoid` | `GlobalExceptionHandler` 返回类型被 springdoc 自动注入 | 类上加 `@Hidden` 从源头阻止扫描 |
| 2 | SSE 端点生成无用类型 | Spring 内部类泄露到 OpenAPI spec | 4 个 SseEmitter 端点加 `@Hidden` |
| 3 | AiController 缺少 Swagger 注解 | 端点全在 "ai-controller" 标签下 | 添加 `@Tag`/`@Operation`/`@Parameter` |
| 4 | 生成的 `BASE_URL` 硬编码 localhost | `OpenAPI.ts` 写死 `http://localhost:8123/api` | 新建 `scripts/fix-openapi-gen.js` 生成后自动 patch |

### 新增/修改文件

| 操作 | 文件 | 说明 |
|------|------|------|
| 新建 | `config/OpenApiConfig.java` | OpenAPI 信息 + 500 响应清理 |
| 修改 | `exception/GlobalExceptionHandler.java` | 添加 `@Hidden` |
| 修改 | `controller/AiController.java` | 添加 `@Tag`/`@Operation`/`@Parameter`；SSE 端点加 `@Hidden` |
| 新建 | `vio-ai-agent-frontend/scripts/fix-openapi-gen.js` | 生成后自动 patch BASE_URL |
| 修改 | `vio-ai-agent-frontend/package.json` | 增加 `openapi:postgen` 步骤 |

### 生成流水线

```
[IDEA 启动后端 :8123]
       │
       ▼
npm run openapi
  ├── openapi:fetch   → curl http://localhost:8123/api/v3/api-docs/default → openapi.json
  ├── openapi:gen     → openapi-typescript-codegen → TypeScript + axios client
  └── openapi:postgen → node scripts/fix-openapi-gen.js (patch BASE_URL + header)
```

### 接口文档访问

- **Swagger UI**：http://localhost:8123/api/swagger-ui.html
- **OpenAPI JSON**：http://localhost:8123/api/v3/api-docs/default

### 技术要点

1. **springdoc 自动扫描 `@RestControllerAdvice` 的返回类型** — 用 `@Hidden` 从源头阻止
2. **SSE 端点不适合代码生成** — 前端使用 `EventSource` 而非 axios，应有独立的手写 wrapper
3. **代码生成后再用脚本 patch** — 避免手动修改生成文件（下次生成会覆盖）
4. **`openapi-typescript-codegen` 已停止维护** — 后续可考虑迁移到 `@hey-api/openapi-ts`
