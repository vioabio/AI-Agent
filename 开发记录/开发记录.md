## 2026-07-01 | 基础通用代码配置

### 一、统一响应包装类

#### 1. `common/Result<T>` — 统一 API 响应体

- `code` / `message` / `data` / `timestamp` 四个字段
- 提供静态工厂方法：`success()`、`error()`
- 支持链式构造，兼容 `ResultCode` 枚举和自定义消息
- 使用 `@JsonInclude(NON_NULL)` 自动忽略空值

#### 2. `common/ResultCode` — 响应状态码枚举

| 分类 | 范围 | 示例 |
|------|------|------|
| HTTP 标准 | 200~503 | SUCCESS、BAD_REQUEST、UNAUTHORIZED、NOT_FOUND、INTERNAL_ERROR |
| 业务异常 | 1xxx | BUSINESS_ERROR(1000)、PARAM_ERROR(1001)、DATA_NOT_FOUND(1002) |
| 系统异常 | 2xxx | SYSTEM_ERROR(2000)、RPC_ERROR(2001)、DB_ERROR(2002) |

---

### 二、自定义异常

#### `exception/BusinessException` — 业务异常基类

- 继承 `RuntimeException`，携带 `code` 和 `message`
- 支持多种构造方式：自定义消息、`ResultCode`、带异常链
- 统一在 `GlobalExceptionHandler` 中捕获处理

---

### 三、全局异常处理器

#### `exception/GlobalExceptionHandler`

使用 `@RestControllerAdvice` 全局拦截异常，覆盖场景：

| 异常类型 | 处理方式 |
|----------|----------|
| `BusinessException` | 返回对应业务状态码和消息 |
| `MethodArgumentNotValidException` | 提取字段校验错误并拼接 |
| `BindException` | 提取表单绑定错误 |
| `ConstraintViolationException` | 提取约束校验错误 |
| `MissingServletRequestParameterException` | 提示缺少必要参数 |
| `MethodArgumentTypeMismatchException` | 提示参数类型错误 |
| `HttpMessageNotReadableException` | 提示请求体格式错误 |
| `HttpRequestMethodNotSupportedException` | 提示请求方法不支持 |
| `MaxUploadSizeExceededException` | 提示上传文件过大 |
| `NoHandlerFoundException` / `NoResourceFoundException` | 404 |
| `Exception` (兜底) | 500，日志记录完整堆栈 |

---

### 四、请求包装类

#### 1. `common/BasePageRequest` — 分页请求基类

- `pageNum`（默认1）、`pageSize`（默认10）、`sortField`、`sortOrder`
- 带 Knife4j `@Schema` 注解

#### 2. `common/PageResult<T>` — 分页响应包装

- `pageNum` / `pageSize` / `total` / `totalPages` / `records`
- 静态工厂 `of()` 自动计算总页数
- `empty()` 返回空分页结果

#### 3. `common/RequestWrapper` — 请求体缓存包装

- 继承 `HttpServletRequestWrapper`，缓存请求体字节
- 支持多次读取 `getInputStream()` / `getReader()`
- 提供 `getBody()` 直接获取请求体字符串
- 用于 Filter 中记录请求日志

#### 4. `common/RequestContext` — 请求上下文

- 基于 `ThreadLocal<Map<String, Object>>` 实现
- 存储 TraceId、UserId、Username、ClientIp、RequestUri
- 请求结束时必须调用 `clear()` 防止内存泄漏

### 一、SDK 接入（`demo/invoke/SdkAiInvoke.java`）

使用阿里云 DashScope 官方 SDK 直接调用：

- `Generation` + `GenerationParam` 构建请求
- 通过 `systemMsg` + `userMsg` 组装对话
- 模型：`qwen-plus`，输出格式：`MESSAGE`
- API Key 引用 `TestApiKey.API_KEY`

**运行方式**：单独运行 `SdkAiInvoke.main()`，不是在 Spring Boot 项目中运行。

---

### 二、HTTP 接入（`demo/invoke/HttpAiInvoke.java`）

通过 Hutool `HttpRequest` 直接请求阿里云 REST API：

- 端点：`https://dashscope.aliyuncs.com/api/v1/services/aigc/text-generation/generation`
- 手动构建 JSON 请求体（`model` + `input.messages` + `parameters`）
- `Authorization: Bearer <apiKey>` 认证

**运行方式**：单独运行 `HttpAiInvoke.main()`。

---

### 三、LangChain4j 接入（`demo/invoke/LangChainAiInvoke.java`）

使用 LangChain4j 社区版 DashScope 适配器：

- `QwenChatModel` 构建器配置 `apiKey` + `modelName("qwen-max")`
- 一行 `chat()` 即可调用，代码最简洁

**运行方式**：单独运行 `LangChainAiInvoke.main()`。

---

### 四、Spring AI 接入（`demo/invoke/SpringAiAiInvoke.java`）⚠ 未激活

通过 Spring AI Alibaba Starter 集成：

- 实现 `CommandLineRunner`，注入 `ChatModel dashscopeChatModel`
- `@Component` 注解**已被注释**，所以项目启动时**不会**自动执行
- 这是唯一一个设计为融入 Spring Boot 生命周期的调用方式

**运行方式**：取消 `@Component` 注释后，启动 Spring Boot 项目即自动执行。

---

### ⚠ 关键注意事项

1. **四种 demo 目前都是独立运行，未集成到 Spring Boot 应用中**。运行 `VioAiAgentApplication` 只会启动 Web 服务器，不会触发任何 AI 调用。要测试 AI 调用必须单独运行各自的 `main()` 方法。

2. **API Key 安全问题**：`TestApiKey.API_KEY` 硬编码在源码中，且已提交到 Git 仓库。建议迁移到 `application.yml` 配置文件并通过 `@Value` 注入，同时将 API Key 添加到 `.gitignore` 或使用环境变量。

3. **Spring AI Alibaba Starter 的 BOM 问题**：版本 `1.0.0-M6.1` 为里程碑版本，可能需要额外配置 Spring Milestone 仓库才能正常解析依赖。

#### 基础提示技巧

| 技巧 | 说明 | 核心要点 |
|------|------|----------|
| 明确指定任务和角色 | 为 AI 提供清晰的任务描述和角色定位 | 帮助模型理解背景和期望 |
| 提供详细说明和具体示例 | 提供足够的上下文信息和期望的输出格式 | 减少模型的不确定性 |
| 使用结构化格式引导思维 | 通过列表、表格等结构化格式 | 使指令更易理解，输出更有条理 |
| 明确输出格式要求 | 指定输出的格式、长度、风格等要求 | 获得更符合预期的结果 |

#### 进阶提示技巧

| 技巧 | 说明 | 适用场景 |
|------|------|----------|
| 思维链提示法（Chain-of-Thought） | 引导模型展示推理过程，逐步思考 | 提高复杂问题的准确性 |
| 少样本学习（Few-Shot Learning） | 提供几个输入-输出对示例 | 帮助模型理解任务模式 |
| 分步骤指导（Step-by-Step） | 将复杂任务分解为可管理的步骤 | 确保模型完成每个关键环节 |
| 自我评估和修正 | 让模型评估自己的输出并改进 | 提高准确性和质量 |
| 知识检索和引用 | 引导模型检索相关信息并明确引用来源 | 提高可靠性 |
| 多视角分析 | 从不同角度、立场或专业视角分析 | 提供全面见解 |
| 多模态思维 | 结合不同表达形式进行思考 | 复杂系统设计 |

**总结**：任务越复杂，就越要给 Prompt 补充更多细节。把 AI 当成人类，模糊的问题得到模糊的答案。

### 一、已完成功能

#### 1. 多轮对话 AI 应用 — `app/LoveApp.java`

使用 Spring AI ChatClient + MessageChatMemoryAdvisor 实现：

- **系统提示词**：扮演恋爱心理专家，围绕单身/恋爱/已婚三种状态引导用户
- **对话记忆**：使用 `MessageWindowChatMemory`（内存存储，最多保留 20 条消息）
- **同步对话** `doChat(message, chatId)`：返回完整响应
- **流式对话** `doChatByStream(message, chatId)`：返回 `Flux<String>` 支持 SSE
- 支持通过 `chatId` 区分不同会话

```java
@Component
public class LoveApp {
    private final ChatClient chatClient;
    private static final String SYSTEM_PROMPT = "扮演深耕恋爱心理领域的专家...";

    public LoveApp(ChatModel dashscopeChatModel) {
        MessageWindowChatMemory chatMemory = MessageWindowChatMemory.builder()
                .chatMemoryRepository(new InMemoryChatMemoryRepository())
                .maxMessages(20).build();
        chatClient = ChatClient.builder(dashscopeChatModel)
                .defaultSystem(SYSTEM_PROMPT)
                .defaultAdvisors(
                    MessageChatMemoryAdvisor.builder(chatMemory).build(),
                    new MyLoggerAdvisor()
                ).build();
    }
}
```

#### 2. 自定义 Advisor — `advisor/`

| 文件 | 功能 | 说明 |
|------|------|------|
| `MyLoggerAdvisor.java` | 日志拦截器 | 同时实现 `CallAroundAdvisor` 和 `StreamAroundAdvisor`，记录每次请求和响应 |
| `ReReadingAdvisor.java` | Re2 推理增强 | 让模型重新阅读问题以提高推理能力（成本加倍，C端不建议使用） |

**Advisor 执行流程**：前置增强（改写 Prompt）→ 调用 ChatModel → 后置增强（记录日志/处理结果）

> `MyLoggerAdvisor` 当前已启用，`ReReadingAdvisor` 已注释（按需开启）。

#### 3. 结构化输出 — 恋爱报告功能

在 `LoveApp` 中定义 `LoveReport` record 并实现 `doChatWithReport()`：

```java
record LoveReport(String title, List<String> suggestions) {}

public LoveReport doChatWithReport(String message, String chatId) {
    return chatClient.prompt()
            .system(SYSTEM_PROMPT + "每次对话后都要生成恋爱结果...")
            .user(message)
            .advisors(spec -> spec.param(ChatMemory.CONVERSATION_ID, chatId))
            .call()
            .entity(LoveReport.class);
}
```

**原理**：Spring AI 将 Java 类转为 JSON Schema 附加到提示词中 → AI 生成符合格式的 JSON → 自动反序列化为 Java 对象。

#### 4. 对话记忆持久化 — `chatmemory/FileBasedChatMemory.java`

实现 `ChatMemory` 接口，使用 **Kryo** 序列化库将对话持久化到磁盘：

- 存储位置：`${user.dir}/tmp/chat-memory/`
- 序列化格式：Kryo 二进制（`.kryo` 文件）
- 解决了 `Message` 接口多子类、无 Serializable 实现的问题

> 当前 LoveApp 使用内存存储，文件持久化已实现但被注释，可按需切换。

### 四、第03章技术要点总结

### 技术要点总结

1. **ETL 流水线顺序决定 RAG 质量上限**：先切割后增强，确保每个切片都能获得精准的关键词元数据

6. **`classpath:` 资源路径优于绝对路径**：确保项目在不同环境、不同机器上均可运行
7. **Spring AI 1.0.0 API 变更**：`toolCallbacks()` → `tools()`，`ChatMemory.CONVERSATION_ID` 替代 `CHAT_MEMORY_CONVERSATION_ID_KEY`

### 二、当前遗留问题

#### 1. `search-api.api-key` 未配置有效值

`application.yml` 中 `search-api.api-key` 当前值为占位文本 `你的 API Key`。需去 [searchapi.io](https://www.searchapi.io) 注册免费账号获取 Key，否则 `WebSearchTool` 调用时返回错误。

```yaml
search-api:
  api-key: 你的 API Key  # ← 需替换为真实 Key
```

#### 2. PDF 中文字体依赖系统字体

`PDFGenerationTool` 使用 `STSongStd-Light`（华文宋体），该字体并非所有系统自带的：
- **Windows**：通常已预装
- **Linux/Docker**：需安装 `fonts-chinese` 或手动放入字体文件

可选方案：将 `src/main/resources/static/fonts/simsun.ttf` 作为备用字体路径，通过 `PdfFontFactory.createFont(filePath, PREFER_EMBEDDED)` 加载。

#### 3. `TerminalOperationTool` Windows 绑定

当前硬编码 `cmd.exe /c`，在 Linux/Mac 下无法执行。解决方案：
- 方案 A：通过 `System.getProperty("os.name")` 判断操作系统自动切换
- 方案 B：Docker 部署时统一使用 Linux 容器（`/bin/sh -c`）

#### 4. `WebSearchTool` 强依赖 SearchAPI 第三方服务

SearchAPI 是付费 SaaS（有免费额度），若服务不可用则搜索功能完全失效。可考虑增加备用搜索引擎（如直接调用 Bing/Google Custom Search API）。

#### 5. 工具调用缺少错误恢复机制

当前工具异常信息以字符串返回给 AI 模型，模型可以基于错误信息调整策略。但如果工具持续失败，没有重试或降级逻辑。后续可在 Agent 阶段（第08章）通过 `ToolCallAgent` 的 ReAct 循环来改善。

#### 6. 工具调用的安全风险

- **`TerminalOperationTool`**：允许 AI 决定执行任意系统命令，存在极高风险。当前仅开发环境使用，生产环境需加命令白名单或禁用
- **`FileOperationTool`**：可读写任意文件，需限制操作目录范围
- **`ResourceDownloadTool`**：可下载任意 URL 资源，需限制文件类型和大小

#### 7. 与后续阶段的耦合点（待完成）

### 四、技术要点总结

1. **`@Tool` + `@ToolParam` 注解式工具定义**是 Spring AI 推荐方式：简洁直观，自动生成 JSON Schema，支持大多数 Java 类型
2. **工具返回值统一使用 String**：AI 模型只能理解文本，返回 String 可确保模型正确解析工具执行结果
3. **`ToolCallbacks.from()` 工厂方法**将注解对象批量转换为 `ToolCallback[]` 数组
4. **`@Autowired(required = false)` 渐进式注入**：工具 Bean 未就绪时系统不受影响，注册后自动激活
5. **工具描述决定 AI 是否调用**：`@Tool(description = "...")` 直接影响 AI 的工具选择决策，描述需准确、具体
6. **`ChatClient.tools()` 即插即用**：在 prompt 链上调用 `.tools(allTools)` 即可让 AI 获得工具能力
7. **安全性由应用程序控制**：AI 只负责决定调用哪个工具及参数，真正执行由应用程序完成，所有操作可控
8. **`TerminateTool` 是 Agent 自主规划的关键**：在第08章中，AI 通过调用 `doTerminate` 工具来主动结束 ReAct 循环
