### 三、Maven Wrapper 打包问题修复

#### 问题现象

执行 `mvnw.cmd package` 时报错：`未能解析此远程名称: 'repo.maven.apache.org'`

#### 根因分析

`mvnw.cmd` (PowerShell) 和 `mvnw` (bash) 使用**不同的哈希算法**来定位缓存的 Maven 分发版：

| 脚本 | 哈希算法 | 缓存路径模式 |
|------|---------|-------------|
| `mvnw.cmd` (PowerShell) | SHA-256 | `wrapper/dists/apache-maven-3.9.16-bin/<sha256>/` |
| `mvnw` (bash) | Java `String.hashCode()` | `wrapper/dists/apache-maven-3.9.16/<hashcode>/` |

Maven 3.9.16 已缓存在 Java hashCode 路径下，但 `mvnw.cmd` 在 SHA-256 路径下找不到 → 触发从 `repo.maven.apache.org` 下载 → DNS 解析失败。

#### 修复方案

1. 计算当前 `distributionUrl` 的 SHA-256 哈希，将缓存 Maven 复制到正确的哈希目录
2. 发现 `JAVA_HOME` 应设为 `D:\IDEA`（IntelliJ 自带的 JDK 21，无空格的路径）
3. 使用 bash 版 `./mvnw` 脚本成功打包（自动使用 Java hashCode 匹配到原始缓存）

#### 最终打包命令

```bash
export JAVA_HOME="D:\\IDEA"
./mvnw -f vio-image-search-mcp-server/pom.xml clean package -DskipTests
```

产出：`vio-image-search-mcp-server/target/vio-image-search-mcp-server-0.0.1-SNAPSHOT.jar` (31 MB)

### 一、已完成功能

#### 1. 多轮对话 AI 应用 — `app/LoveApp.java`

使用 Spring AI ChatClient + MessageChatMemoryAdvisor 实现：

- **系统提示词**：扮演恋爱心理专家，围绕单身/恋爱/已婚三种状态引导用户
- **对话记忆**：使用 `MessageWindowChatMemory`（内存存储，最多保留 20 条消息）
- **同步对话** `doChat(message, chatId)`：返回完整响应
- **流式对话** `doChatByStream(message, chatId)`：返回 `Flux<String>` 支持 SSE
- 支持通过 `chatId` 区分不同会话

```java
@Component
public class LoveApp {
    private final ChatClient chatClient;
    private static final String SYSTEM_PROMPT = "扮演深耕恋爱心理领域的专家...";

    public LoveApp(ChatModel dashscopeChatModel) {
        MessageWindowChatMemory chatMemory = MessageWindowChatMemory.builder()
                .chatMemoryRepository(new InMemoryChatMemoryRepository())
                .maxMessages(20).build();
        chatClient = ChatClient.builder(dashscopeChatModel)
                .defaultSystem(SYSTEM_PROMPT)
                .defaultAdvisors(
                    MessageChatMemoryAdvisor.builder(chatMemory).build(),
                    new MyLoggerAdvisor()
                ).build();
    }
}
```

#### 2. 自定义 Advisor — `advisor/`

| 文件 | 功能 | 说明 |
|------|------|------|
| `MyLoggerAdvisor.java` | 日志拦截器 | 同时实现 `CallAroundAdvisor` 和 `StreamAroundAdvisor`，记录每次请求和响应 |
| `ReReadingAdvisor.java` | Re2 推理增强 | 让模型重新阅读问题以提高推理能力（成本加倍，C端不建议使用） |

**Advisor 执行流程**：前置增强（改写 Prompt）→ 调用 ChatModel → 后置增强（记录日志/处理结果）

> `MyLoggerAdvisor` 当前已启用，`ReReadingAdvisor` 已注释（按需开启）。

#### 3. 结构化输出 — 恋爱报告功能

在 `LoveApp` 中定义 `LoveReport` record 并实现 `doChatWithReport()`：

```java
record LoveReport(String title, List<String> suggestions) {}

public LoveReport doChatWithReport(String message, String chatId) {
    return chatClient.prompt()
            .system(SYSTEM_PROMPT + "每次对话后都要生成恋爱结果...")
            .user(message)
            .advisors(spec -> spec.param(ChatMemory.CONVERSATION_ID, chatId))
            .call()
            .entity(LoveReport.class);
}
```

**原理**：Spring AI 将 Java 类转为 JSON Schema 附加到提示词中 → AI 生成符合格式的 JSON → 自动反序列化为 Java 对象。

#### 4. 对话记忆持久化 — `chatmemory/FileBasedChatMemory.java`

实现 `ChatMemory` 接口，使用 **Kryo** 序列化库将对话持久化到磁盘：

- 存储位置：`${user.dir}/tmp/chat-memory/`
- 序列化格式：Kryo 二进制（`.kryo` 文件）
- 解决了 `Message` 接口多子类、无 Serializable 实现的问题

> 当前 LoveApp 使用内存存储，文件持久化已实现但被注释，可按需切换。

### 四、第03章技术要点总结

1. **ChatClient 优于 ChatModel**：支持链式调用、默认系统提示词、Advisor 拦截器、结构化输出
2. **Advisor 即拦截器**：责任链模式，前置/后置增强，可自定义日志、鉴权、Prompt 改写
3. **MessageChatMemoryAdvisor 是首选**：保留原始对话结构，更符合现代 LLM 对话模型设计
4. **结构化输出原理**：调用前附加 JSON Schema，调用后将 JSON 转为 Java 对象
5. **Kryo 解决序列化难题**：Message 接口多子类、无 Serializable，Kryo 是合适方案
6. **PromptTemplate 实现前后端分离**：提示词从代码抽离到文件，便于版本管理
7. **MVP 策略**：先核心对话，再逐步扩展知识库、工具调用、MCP 等高级能力
8. **Token 成本意识**：系统提示词、用户提示词和 AI 输出都消耗成本，需持续优化

### 四、技术要点总结

1. **`@Tool` + `@ToolParam` 注解式工具定义**是 Spring AI 推荐方式：简洁直观，自动生成 JSON Schema，支持大多数 Java 类型
2. **工具返回值统一使用 String**：AI 模型只能理解文本，返回 String 可确保模型正确解析工具执行结果
3. **`ToolCallbacks.from()` 工厂方法**将注解对象批量转换为 `ToolCallback[]` 数组
4. **`@Autowired(required = false)` 渐进式注入**：工具 Bean 未就绪时系统不受影响，注册后自动激活
5. **工具描述决定 AI 是否调用**：`@Tool(description = "...")` 直接影响 AI 的工具选择决策，描述需准确、具体
6. **`ChatClient.tools()` 即插即用**：在 prompt 链上调用 `.tools(allTools)` 即可让 AI 获得工具能力
7. **安全性由应用程序控制**：AI 只负责决定调用哪个工具及参数，真正执行由应用程序完成，所有操作可控
8. **`TerminateTool` 是 Agent 自主规划的关键**：在第08章中，AI 通过调用 `doTerminate` 工具来主动结束 ReAct 循环

### 二、MCP 调用本质

MCP 调用的本质是**工具调用的标准化**：

1. MCP Server 启动后，通过 MCP 协议向 Client 宣告自己提供了哪些工具
2. 主应用的 `ToolCallbackProvider` 从所有 MCP Server 收集工具列表
3. 用户发送消息时，AI 看到本地工具 + MCP 工具的全部能力
4. AI 决定调用某个 MCP 工具 → 主应用通过 MCP 协议向 MCP Server 发起调用 → MCP Server 执行并返回结果 → 主应用将结果返回给 AI → AI 总结回复

**不是 AI 服务器直接调用 MCP 服务**，而是 AI 告诉后端程序"我想调用哪个工具"，后端程序代为执行。

- **信息不对称**：用户只能看到工具基本描述，AI 能看到完整描述（包括隐藏的恶意指令）
- **上下文混合攻击**：恶意 MCP 工具的描述可能影响其他正常工具的行为
- **缺乏权限验证**：MCP 工具被调用时无二次确认机制

**建议防护措施**：
- 在 Docker 沙箱中运行第三方 MCP 服务
- 仅安装来自官方或知名组织的 MCP 服务
- 期望 MCP 官方后续完善权限控制和版本验证机制

### 五、技术要点总结

1. **MCP 本质是工具调用的标准化**：不是新技术，而是让工具可以跨语言、跨平台共享的协议规范
2. **慎用 MCP**：如果不需要共享的工具，直接使用本地 `@Tool` 注解更简单。先开发工具调用，需要共享时再转换为 MCP 服务
3. **STDIO vs SSE 选择**：本地开发优先 STDIO（安全、零网络开销），远程部署/多客户端共享选择 SSE
4. **`MethodToolCallbackProvider` 实现工具到 MCP 的零代码转换**：已有的 `@Tool` 注解工具类无需任何修改即可注册为 MCP 服务
5. **`ToolCallbackProvider` 提供统一的工具门面**：聚合本地工具和所有 MCP Server 工具，AI 无需感知工具来源
6. **`@Autowired(required = false)` 保证渐进式可用**：MCP Server 未就绪时应用不受影响，就绪后自动激活
7. **Windows 命令注意 `.cmd` 后缀**：`mcp-servers.json` 中 `npx` 需写为 `npx.cmd`
8. **STDIO 模式禁止 `System.out.println`**：标准输出流用于 JSON-RPC 通信，任何额外输出都会干扰协议

### 一、开发流程

#### 步骤 1：创建 `AgentState` 状态枚举 — `agent/model/AgentState.java`

```java
public enum AgentState {
    IDLE,       // 空闲 — Agent 尚未开始或已执行完毕
    RUNNING,    // 运行中 — Agent 正在执行 Agent Loop
    FINISHED,   // 已完成 — 任务正常完成（调用了 terminate 或达到最大步数）
    ERROR       // 错误 — 执行过程中发生异常
}
```

状态转换规则：`IDLE → RUNNING → FINISHED` 或 `IDLE → RUNNING → ERROR`。从非 IDLE 状态再次调用 `run()` 会抛出异常，保证每次执行的状态一致性。

#### 步骤 2：创建 `BaseAgent` 抽象基类 — `agent/BaseAgent.java`

智能体框架的核心基类，提供所有智能体的公共基础设施：

| 属性 | 类型 | 说明 |
|------|------|------|
| `name` | String | 智能体名称 |
| `systemPrompt` | String | 系统提示词，定义角色和能力边界 |
| `nextStepPrompt` | String | 每步执行前追加的引导提示词，促使 AI 主动规划 |
| `state` | AgentState | 当前状态（初始 IDLE） |
| `currentStep` | int | 当前步数（从 1 开始） |
| `maxSteps` | int | 最大步数限制（默认 10，安全阀防止无限循环） |
| `chatClient` | ChatClient | AI 对话客户端（由子类或调用方注入，不写死大模型） |
| `messageList` | List\<Message\> | 消息上下文列表（自主维护，不依赖 Spring AI 的 ChatMemory） |

**两个核心执行方法**：

- **`run(String userPrompt)`** — 同步执行，返回所有步骤结果的拼接字符串
- **`runStream(String userPrompt)`** — SSE 流式执行，通过 `CompletableFuture.runAsync()` 异步运行，每步结果实时推送到 `SseEmitter`（5 分钟超时）

**Agent Loop 核心逻辑**（`run()` 方法）：

```
1. 状态校验（必须是 IDLE）
2. state = RUNNING，记录用户消息
3. for (i = 0; i < maxSteps && state != FINISHED; i++)
      step() → 收集结果
4. 超步数检测 → state = FINISHED
5. 异常捕获 → state = ERROR
6. finally: cleanup() 清理资源
```

> **设计要点**：`chatClient` 由调用方通过 setter 注入，不写死大模型；`messageList` 自主维护而不依赖 Spring AI 的 ChatMemory；`step()` 作为模板方法中的可变部分交给子类实现。

#### 步骤 3：创建 `ReActAgent` 抽象类 — `agent/ReActAgent.java`

实现 ReAct（Reasoning + Acting）模式，重写父类 `step()` 方法：

```java
@Override
public String step() {
    boolean shouldAct = think();   // 思考：需要行动吗？
    if (!shouldAct) return "思考完成 - 无需行动";
    return act();                   // 行动：执行工具调用
}
```

两个抽象方法交给子类实现：
- `think()` → 调用 AI 分析上下文，决定是否需要/需要哪些工具
- `act()` → 执行 AI 决定的工具调用，返回执行结果

#### 步骤 4：创建 `ToolCallAgent` — `agent/ToolCallAgent.java`

本阶段最核心的类，具体实现了 `think()` 和 `act()` 方法，可作为具体智能体实例的直接父类。

**构造器关键配置**：

```java
public ToolCallAgent(ToolCallback[] availableTools) {
    this.availableTools = availableTools;
    this.toolCallingManager = ToolCallingManager.builder().build();
    // 关键：禁用 Spring AI 内置工具调用，自主管理消息上下文和执行流程
    this.chatOptions = DashScopeChatOptions.builder()
            .withInternalToolExecutionEnabled(false)
            .build();
}
```

**`think()` 执行流程**：

```
1. 将 nextStepPrompt 作为 UserMessage 追加到 messageList
2. 调用 ChatClient.prompt(messageList)
        .system(systemPrompt)
        .tools(availableTools)
        .call()
3. 解析 AssistantMessage，提取 ToolCall 列表
4. 有 ToolCall → 保存 ChatResponse，返回 true（进入 act）
5. 无 ToolCall → 记录纯文本回复到 messageList，返回 false（Agent Loop 继续下一轮）
```

**`act()` 执行流程**：

```
1. 通过 ToolCallingManager.executeToolCalls() 执行 AI 选择的工具
2. 用 conversationHistory 替换 messageList（含助手消息 + 工具返回消息）
3. 检查是否调用了 doTerminate → 若是，state = FINISHED
4. 返回工具执行结果描述
```

> **为什么要禁用内置工具执行？** Spring AI 默认的 `ChatClient` 工具调用模式是"一次请求-一次工具调用-一次返回"，适合 LoveApp 的简单对话场景。但 Agent 需要在一个循环中多次调用 AI + 多次执行工具，消息上下文必须由 Agent 自主管理，因此必须设置 `withInternalToolExecutionEnabled(false)`。

#### 步骤 5：创建 `VioManus` 超级智能体 — `agent/VioManus.java`

使用 `@Component` 注册为 Spring Bean，注入 `ToolCallback[]` 和 `ChatModel`：

```java

        super(allTools);
        this.setName("VioManus");
        this.setSystemPrompt(SYSTEM_PROMPT);    // 英文系统提示词

                .defaultAdvisors(new MyLoggerAdvisor())
                .build();

    }

```

**系统提示词设计要点**：
- 使用英文（AI 模型对英文指令的遵循度更高）
- 明确列出所有可用工具及其用途
- 强调"step by step"和"主动推进任务"

**下一步提示词设计要点**：
- 引导 AI 主动选择工具、分解任务
- 告知 AI：简单问题直接回答，不需要工具
- **最关键**：告知 AI 何时调用 `terminate` 结束任务（没有这一步 Agent 只会被动达到 maxSteps）

### 四、当前遗留问题

#### 1. Token 消耗巨大

Agent 每执行一步至少消耗一次 AI 调用（Think 阶段），多步任务的 Token 消耗是普通对话的数倍甚至数十倍：

- **简单问题**（1 步）：约 500-1000 tokens
- **中等任务**（3-5 步）：约 3000-8000 tokens
- **复杂任务**（10+ 步）：可能超过 20000 tokens

**优化方向**：
- 在简单对话场景下优先使用 LoveApp（单次调用），而非 VioManus（多步循环）
- 通过 `maxSteps` 限制上限（当前 20 步）
- 后续可增加"任务复杂度判断"：简单问题直接路由到 LoveApp，复杂问题才启动 Agent

#### 2. 无限循环风险

尽管有 `maxSteps` 安全阀，AI 可能在某些情况下陷入"无效循环"：
- 工具持续失败 → AI 不断重试 → 达到 maxSteps 被迫终止
- AI 忘记调用 `terminate` → 每步都给文本回复（不调工具）→ Agent Loop 空转

**当前缓解措施**：`maxSteps=20` + `nextStepPrompt` 明确告知 AI 何时调 `terminate`。

**后续优化**：可增加"卡住检测"（连续 N 步无进展则主动终止）和"重复检测"（连续调用相同工具相同参数则警告）。

#### 3. Agent 无长期记忆

当前 `messageList` 仅在单次 `run()` 调用中有效，Agent 执行完毕后消息全部丢弃。用户无法在同一会话中连续与 Agent 交互（每次都是全新的上下文）。

**对比 LoveApp**：LoveApp 通过 `MessageChatMemoryAdvisor` 实现了多轮对话记忆，但 Agent 选择了自主管理消息的路径。后续可考虑在 BaseAgent 中集成 ChatMemory。

#### 4. 工具调用缺乏并发能力

`ToolCallAgent.think()` 可以返回多个 ToolCall，但 `act()` 中的 `ToolCallingManager.executeToolCalls()` 的并发行为由 Spring AI 内部决定。如果两个工具之间无依赖关系（如同时搜索两个关键词），串行执行会浪费时间。

#### 5. 无 MCP 工具集成

当前 VioManus 只注入了 `ToolCallback[] allTools`（本地 7 个工具），未集成 MCP `ToolCallbackProvider`（远程 MCP 工具）。后续可在 `VioManus` 构造器中同时接入两种工具源。

#### 6. 无 Agent 工作流编排

当前只有一个 VioManus 单体智能体。开发文档中描述了 5 种多智能体工作流模式（Prompt Chaining、Routing、Parallelization、Orchestrator-Workers、Evaluator-Optimizer），当前均未实现。

#### 7. 与后续阶段的耦合点

| 依赖项 | 所属阶段 | 当前状态 |
|--------|----------|----------|
| MCP 工具集成到 Agent | 第07章→第08章 | 🔗 可扩展：在 VioManus 构造器中注入 `ToolCallbackProvider` |
| 多智能体工作流 | 第08章扩展 | ❌ 未实现：需引入 Spring AI Alibaba Graph 或 LangGraph |

### 六、技术要点总结

1. **模板方法模式是 Agent 框架的核心**：BaseAgent 定义 Agent Loop 骨架，子类通过 `step()`/`think()`/`act()` 填充具体行为
2. **`withInternalToolExecutionEnabled(false)` 是 Agent 与普通 ChatClient 的本质区别**：Agent 需要在循环中多次调用 AI + 多次执行工具，必须自主管理消息上下文
3. **`doTerminate` 工具是 Agent 自主终止的唯一正常途径**：没有它，Agent 只能被动等待 maxSteps 耗尽。提示词中必须明确告知 AI 何时调用
4. **每次请求必须创建新的 Agent 实例**：Agent 维护内部状态（`messageList`、`state`），跨请求共享会导致状态冲突
5. **`nextStepPrompt` 是引导 AI 主动规划的关键**：每步 Think 前都会追加此提示词，促使 AI 不断推进任务进度
6. **`maxSteps` 是成本和安全的平衡点**：步数太少复杂任务无法完成，步数太多 Token 浪费严重。当前 20 步适合大多数场景
7. **Agent 没有长期记忆**：与 LoveApp 不同，Agent 执行完毕后消息全部丢弃。简单多轮对话应使用 LoveApp，复杂自主任务才用 VioManus
8. **ReAct 模式的本质是"思考-行动-观察"循环**：每次工具执行的结果自动成为下一轮思考的上下文，形成闭环

#### 步骤 2：编写前端核心文件

**2A. `src/api/index.js` — 前后端通信桥梁**（最关键的文件）

```js
const API_BASE_URL = import.meta.env.MODE === 'production'
  ? '/api'                         // 生产：通过 Nginx 反向代理到后端
  : 'http://localhost:8123/api'    // 开发：CORS 直连后端

// 通用 SSE 连接函数（基于 EventSource API）
function connectSSE(url, params, onMessage, onError) {
  const queryString = Object.entries(params)
    .map(([k, v]) => `${encodeURIComponent(k)}=${encodeURIComponent(v)}`)
    .join('&')
  const fullUrl = `${API_BASE_URL}${url}?${queryString}`
  const eventSource = new EventSource(fullUrl)
  eventSource.onmessage = (event) => {
    if (event.data === '[DONE]') { eventSource.close(); return }
    onMessage(event.data)
  }
  eventSource.onerror = (error) => { eventSource.close(); if (onError) onError(error) }
  return eventSource
}

export function chatWithLoveApp(message, chatId, onMessage, onError) {
  return connectSSE('/ai/love_app/chat/sse', { message, chatId }, onMessage, onError)
}

export function chatWithManus(message, onMessage, onError) {
  return connectSSE('/ai/manus/chat', { message }, onMessage, onError)
}
```

**设计要点**：
- 使用浏览器原生 `EventSource` API（GET 请求，自动重连），不需要 axios
- 开发/生产环境通过 `import.meta.env.MODE` 自动切换 API 地址
- SSE 仅支持 GET，后端端点必须用 `@GetMapping`

**2B. `src/router/index.js` — 3 条路由**

| 路径 | 组件 | 标题 |
|------|------|------|
| `/` | Home.vue | VIO AI Agent |
| `/love-master` | LoveMaster.vue | AI 恋爱大师 |
| `/super-agent` | SuperAgent.vue | AI 超级智能体 |

- 使用 `createWebHistory()`（HTML5 History 模式），需要 Nginx `try_files` 回退
- 路由懒加载（`() => import(...)`），按需加载页面组件
- 全局 `beforeEach` 守卫动态设置 `document.title`

**2C. `src/views/Home.vue` — 首页**

两张渐变卡片（赛博朋克风格），点击导航到聊天页面：
- 💕 AI 恋爱大师 → `/love-master`
- 🤖 AI 超级智能体 → `/super-agent`

**2D. `src/views/LoveMaster.vue` — 恋爱大师聊天页**

- 进入页面自动生成 `chatId`（`love_` + 8 位随机字符）
- 调用 `chatWithLoveApp()` 发起 SSE 连接
- SSE 数据**增量追加**到同一个消息气泡（流式打字效果）
- 页面卸载时 `onBeforeUnmount` 关闭 EventSource

**2E. `src/views/SuperAgent.vue` — 超级智能体聊天页**

- 调用 `chatWithManus()` 发起 SSE 连接
- SSE 数据**智能拆分**为多个消息气泡：
  - 按中文标点 `。！？…` 拆分
  - 缓冲超过 40 字符时拆分
  - 气泡间最小间隔 800ms（模拟自然思考节奏）
- Agent 每步执行结果显示为独立气泡
- 顶部状态栏实时显示：就绪 / 思考中 / 执行中 / 已完成

**2F. `src/components/ChatRoom.vue` — 聊天室核心组件**

- Props：`messages`（消息数组）、`connectionStatus`（'idle'/'connecting'/'done'）、`aiType`（'love'/'super'）
- AI 消息左对齐 + 头像，用户消息右对齐 + emoji 头像
- 连接中显示三点跳动动画（"思考中..."）
- 新消息自动滚动到底部（`watch` + `nextTick`）
- Textarea 输入框，Enter 发送，按钮发送
- 连接中禁用输入

#### 步骤 3：编写后端 Dockerfile（运行时打包方式）

在项目根目录新建 `Dockerfile`，使用 Maven 容器内编译（无需手动打 jar）：

```dockerfile
FROM maven:3.9-amazoncorretto-21
WORKDIR /app
COPY pom.xml .
COPY .mvn .mvn
COPY mvnw .
COPY mvnw.cmd .
RUN mvn dependency:go-offline -B -q || true
COPY src ./src
RUN mvn clean package -DskipTests -q
EXPOSE 8123
CMD ["java", "-jar", "/app/target/vio-ai-agent-0.0.1-SNAPSHOT.jar", "--spring.profiles.active=prod"]
```

**Docker 层缓存优化**：先复制 pom.xml + mvnw，下载依赖，再复制 src 编译。源码修改时只需重新编译，不需要重新下载依赖。

#### 步骤 4：编写前端 Dockerfile（多阶段构建）

```dockerfile
FROM node:20-alpine AS build       # Stage 1: Node 编译
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build

FROM nginx:alpine                   # Stage 2: Nginx 托管
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**两阶段分离的好处**：最终镜像只有 Nginx + 静态文件（~20MB），不包含 Node.js 和 node_modules。

#### 步骤 5：编写 nginx.conf

Nginx 配置承担三个职责：

| 职责 | 配置项 | 关键技术点 |
|------|--------|-----------|
| **SPA 路由回退** | `try_files $uri $uri/ /index.html` | Vue Router History 模式，刷新页面不 404 |
| **API 反向代理** | `location ^~ /api/ { proxy_pass ... }` | 前端 `/api/` 请求转发到后端服务 |
| **SSE 流式支持** | `proxy_buffering off` + `chunked_transfer_encoding off` | 🔑 关闭 Nginx 缓冲，实现实时推送 |
| **长连接超时** | `proxy_read_timeout 600s` | 🔑 Agent 执行可能长达几分钟 |
| **静态资源缓存** | `expires 1y` | js/css/图片长期缓存 |

> **为什么 SSE 必须关闭缓冲？** Nginx 默认会缓冲后端响应，收集到一定大小后再发给客户端。SSE 需要逐条实时推送，如果开启缓冲，用户会在 Agent 全部执行完毕后一次性看到所有结果。

### 三、前后端数据流

```

  Browser :3000 ──CORS──> Spring Boot :8123/api

生产环境 (Docker)
  Browser :80 → Nginx (前端容器)
```

### 六、技术要点总结

1. **SSE 单向流式是 Agent 分步输出的最佳方案**：浏览器 `EventSource` 天然支持、自动重连、无需 WebSocket 的复杂性
2. **`EventSource` 只用 GET，后端端点必须 `@GetMapping`**：POST 请求无法使用 SSE
3. **Nginx `proxy_buffering off` 是 SSE 的生命线**：开启缓冲会积压所有数据直到连接关闭才推送，实时性完全丧失
4. **多阶段 Docker 构建减小镜像体积**：前端最终镜像只含 Nginx + 静态文件（~20MB），编译所需的 Node.js 不进入最终镜像
5. **`import.meta.env.MODE` 实现环境自动切换**：开发直连后端、生产走 Nginx 代理，无需手动改配置
6. **Vue Router History 模式需要服务端配合**：Nginx `try_files $uri $uri/ /index.html` 确保刷新页面不 404
7. **Docker 层缓存优化**：Dockerfile 中先 COPY pom.xml 下载依赖，再 COPY src 编译，源码修改时复用缓存的依赖层
8. **前端两种气泡策略对应两种后端流式模式**：LoveMaster 增量追加（`Flux<String>` 逐 token），SuperAgent 智能拆分（`SseEmitter` 逐 step）
