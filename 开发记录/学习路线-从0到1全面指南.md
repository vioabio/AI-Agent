#### 1.2 面试题与参考答案

**Q1: 为什么要设计统一的响应体 `Result<T>`？有什么好处？**

**答**：统一响应体的核心价值在于**契约一致性**。具体好处包括：
1. **前端解析统一**：前端只需处理一种 JSON 结构 `{code, message, data, timestamp}`，无需为不同接口写不同的解析逻辑
2. **错误处理集中化**：通过 `GlobalExceptionHandler` 统一捕获所有异常并包装为 `Result`，避免异常信息直接暴露给客户端
3. **可扩展性**：`code` 字段可以区分 HTTP 标准码和业务码（如 1000=业务异常，2000=系统异常），便于监控和排查
4. **泛型设计**：`Result<T>` 支持不同数据类型，`@JsonInclude(NON_NULL)` 自动忽略 null 字段使响应更简洁
5. **链路追踪**：`timestamp` 字段帮助定位问题发生时间，后续可扩展 `traceId` 实现分布式链路追踪

**Q2: `@RestControllerAdvice` 和 `@ControllerAdvice` 的区别是什么？全局异常处理器的执行顺序是怎样的？**

**答**：
- `@RestControllerAdvice` = `@ControllerAdvice` + `@ResponseBody`，方法返回值自动序列化为 JSON，适用于 REST API
- `@ControllerAdvice` 返回值会走视图解析器，适用于服务端渲染
- 执行顺序：Spring 按 `@Order` 或 `@Priority` 排序多个异常处理器，匹配优先级为**子类异常优先于父类异常**。例如先匹配 `BusinessException` 的 handler，未匹配则 fallback 到 `Exception` 的兜底 handler

**Q3: `ThreadLocal` 在使用时需要注意什么问题？**

**答**：
1. **内存泄漏**：`ThreadLocal` 的 key 是弱引用，但 value 是强引用。线程池场景下线程复用，如果不调用 `remove()`，value 永远不会被 GC。解决方案：请求结束后在 `finally` 块或拦截器的 `afterCompletion` 中调用 `clear()`
2. **线程传递**：父线程设置的 `ThreadLocal` 子线程拿不到，需要 `InheritableThreadLocal` 或手动传递
3. **异步场景**：`@Async` 或 `CompletableFuture` 会导致线程切换，需要显式传递上下文

**Q4: 什么是 CORS？Spring Boot 如何配置跨域？**

**答**：
CORS（Cross-Origin Resource Sharing）是浏览器安全策略。当页面从 `localhost:3000`（前端）请求 `localhost:8123`（后端）时，浏览器会发送预检请求（OPTIONS）验证服务器是否允许跨域。

Spring Boot 配置方式：
1. **全局配置**：实现 `WebMvcConfigurer.addCorsMappings()`，设置 `allowedOriginPatterns`、`allowedMethods`、`allowedHeaders`、`allowCredentials`、`maxAge`
2. **过滤器配置**：注册 `CorsFilter` Bean，确保在 Spring Security Filter 之前执行（`@Order(HIGHEST_PRECEDENCE)`）
3. **注解配置**：`@CrossOrigin` 标注在 Controller 或方法上（粒度最细）

#### 2.2 面试题与参考答案

**Q1: 什么是 Token？为什么大模型按 Token 计费？**

**答**：
- **Token** 是大模型理解和生成文本的最小语义单元。英文中约 1 Token ≈ 0.75 个单词（约 4 个字符），中文中约 1 字 ≈ 2 Token
- **按 Token 计费的原因**：Token 数量决定了模型的计算量。每次推理时模型需要对每个 Token 执行矩阵运算（主要是注意力机制的计算），Token 越多 → 计算量越大 → GPU 资源消耗越大 → 成本越高
- 成本公式：总成本 = (输入 Token 数 × 输入单价) + (输出 Token 数 × 输出单价)，输出通常比输入贵 2-4 倍

**Q2: 什么是 Context Window（上下文窗口）？有什么限制和优化策略？**

**答**：
- **Context Window** 是模型一次能处理的最大 Token 数，包括 System Prompt + 历史对话 + 用户输入 + 模型输出
- **限制**：超出窗口的内容会被截断（通常截断最早的对话），导致模型"失忆"
- **优化策略**：
  1. **滑动窗口记忆**：只保留最近 N 条消息（如 `MessageWindowChatMemory.maxMessages(20)`）
  2. **摘要压缩**：对早期对话生成摘要，用摘要代替原文
  3. **向量检索记忆**：只检索与当前问题最相关的历史消息
  4. **工具输出截断**：`MAX_TOOL_OUTPUT_LENGTH = 3000`，防止工具结果塞爆窗口

**Q3: Spring AI 的 `ChatClient` 和 `ChatModel` 有什么区别？为什么要用 `ChatClient`？**

**答**：
- **`ChatModel`**：底层接口，直接调用大模型，需要手动构建 `Prompt` 对象，没有 Advisor 拦截器、ChatMemory 等高级特性
- **`ChatClient`**：高级封装，提供流式 API（`.stream().content()`）、链式调用、默认 System Prompt、Advisor 拦截器链、结构化输出（`.entity(Class)`）、工具调用（`.tools()`/`.toolCallbacks()`）
- **选择**：`ChatClient` 是 Spring AI 推荐的编程模型，功能更丰富，代码更简洁

**Q4: 什么是思维链（Chain-of-Thought）提示？为什么能提高准确率？**

**答**：
- **定义**：在提示词中要求模型"逐步思考"、"展示推理过程"，引导模型将复杂问题分解为中间推理步骤
- **原理**：大模型是自回归的（每个 Token 基于前面的所有 Token 生成）。当模型先生成推理步骤时，这些步骤本身成为了后续生成的上下文条件，相当于给模型更多"思考空间"
- **经典句式**："Let's think step by step"
- **局限**：增加输出 Token（推理过程也计费），简单任务不需要

#### 3.2 面试题与参考答案

**Q1: 什么是 Advisor？它在 Spring AI 中的作用是什么？**

**答**：
- **Advisor** 是 Spring AI 中的**拦截器/中间件**模式实现，采用**责任链（Chain of Responsibility）**设计模式
- **作用**：在 ChatClient 调用 AI 模型的前后执行增强逻辑
  - **前置增强**（before）：改写用户 Prompt、注入上下文（如 `MessageChatMemoryAdvisor` 注入历史消息）
  - **后置增强**（after）：记录日志、处理响应、提取结构化数据
- **内置 Advisor**：
  - `MessageChatMemoryAdvisor`：自动管理对话记忆
  - `QuestionAnswerAdvisor`：RAG 检索增强
  - `ContextualQueryAugmenter`：上下文查询增强
- **自定义**：实现 `CallAroundAdvisor`（同步）和 `StreamAroundAdvisor`（流式）

**Q2: SSE 和 WebSocket 有什么区别？为什么 AI 流式输出选 SSE？**

**答**：

| 维度 | SSE | WebSocket |
|------|-----|-----------|
| 通信方向 | 单向（服务端→客户端） | 双向（全双工） |
| 协议 | HTTP（标准） | 独立协议（ws://） |
| 自动重连 | EventSource 内置 | 需手动实现 |
| 代理兼容 | 完全兼容 HTTP 代理 | 部分代理不支持 |
| 复杂度 | 极低（浏览器原生 API） | 需额外库 |

**选择 SSE 的原因**：AI 对话场景是用户发送一条消息 → 服务端持续返回（客户端不主动发数据），SSE 更合适。WebSocket 虽然双向，但需要额外处理心跳、重连、序列化，过度设计。

**Q3: 结构化输出（Structured Output）的原理是什么？**

**答**：
1. **定义 Java Record/Class**：如 `record LoveReport(String title, List<String> suggestions) {}`
2. **Spring AI 自动生成 JSON Schema**：反射 Record 的字段类型，生成对应的 JSON Schema（如 `{"type": "object", "properties": {"title": {"type": "string"}, ...}}`）
3. **附加到 Prompt**：JSON Schema 作为输出格式约束，告诉模型"请按这个格式输出 JSON"
4. **模型返回 JSON**：模型生成符合 Schema 的 JSON 字符串
5. **反序列化**：Spring AI 使用 Jackson 将 JSON 反序列化为 Java 对象

**Q4: 为什么 `Message` 接口的序列化是个难题？Kryo 如何解决？**

**答**：
- **难题**：`Message` 是 Spring AI 的核心接口，有多个子类（`UserMessage`、`AssistantMessage`、`ToolResponseMessage` 等），这些类没有实现 `Serializable`，且其内部结构含 `Map<String, Object>` 等复杂类型，Jackson 标准序列化无法处理多态
- **Kryo 解决方案**：
  1. `setRegistrationRequired(false)`：允许序列化未注册的类
  2. `setInstantiatorStrategy(new StdInstantiatorStrategy())`：使用 Objenesis 绕过无参构造器要求，直接通过字节码创建对象
  3. Kryo 原生支持多态序列化/反序列化，不需要类实现 `Serializable`

#### 4.2 面试题与参考答案

**Q1: 什么是 RAG？为什么需要 RAG？**

**答**：
- **RAG**（Retrieval-Augmented Generation，检索增强生成）是一种将**信息检索**与**大模型生成**结合的技术。流程：用户提问 → 从知识库检索相关文档 → 将检索结果与问题一起提交给 LLM → LLM 基于检索内容生成回答
- **为什么需要**：
  1. **知识时效性**：LLM 的训练数据有截止日期，RAG 可以接入最新文档
  2. **幻觉问题**：LLM 可能编造不存在的信息，RAG 约束模型基于真实文档回答，并可通过 `emptyContextPromptTemplate` 在无匹配时拒绝回答
  3. **私有知识**：企业内部文档无法用于训练公开模型，RAG 实现私有知识的安全使用
  4. **可追溯性**：可以明确告知用户答案来源于哪些文档

**Q2: 什么是 Embedding？为什么选择 1536 维？**

**答**：
- **Embedding** 是将文本映射到高维向量空间的过程。语义相近的文本在向量空间中距离相近
- **1536 维**：这是 DashScope Embedding 模型的输出维度（对应 OpenAI text-embedding-ada-002 的维度）
- **维度选择权衡**：
  - 维度越高 → 表达能力越强 → 存储和检索成本越高
  - 1536 是行业事实标准，兼容性最好
- **应用**：`similaritySearch(queryEmbedding)` 在向量空间中查找与 query 最相似的文档向量

**Q3: 为什么文档需要切割（Chunking）？Chunk Size 和 Overlap 如何选择？**

**答**：
- **为什么切割**：
  1. **上下文窗口限制**：LLM 的 Context Window 有限，完整文档可能太长
  2. **检索精度**：小块文档更精准匹配具体问题，大块文档包含太多无关信息
  3. **成本**：注入 Prompt 的文档也消耗 Token，小块更经济
- **参数选择**：
  - `chunkSize=200` Token：每块约 200 Token，适合精准检索
  - `overlap=100` Token：相邻块之间重叠 100 Token，防止关键信息被切在边界处丢失
  - 经验法则：overlap = chunkSize 的 10%-50%

**Q4: RAG 的 ETL 流程是什么？各阶段的作用？**

**答**：
ETL = Extract → Transform → Load：
1. **Extract（抽取）**：`DocumentLoader` 从数据源（文件系统、数据库、API）加载原始文档，提取文本内容和元数据（如"单身/恋爱/已婚"标签）
2. **Transform（转换）**：对文档进行加工
   - `TokenTextSplitter`：按 Token 切割文档
   - `KeywordMetadataEnricher`：用 AI 提取关键词，增强元数据（提升检索精度）
3. **Load（加载）**：将处理后的文档向量化（调用 Embedding API）并写入 `VectorStore`

**Q1: PGVector 和 SimpleVectorStore 的区别是什么？什么场景下用哪个？**

**答**：

| 维度 | SimpleVectorStore | PGVector |
|------|-------------------|----------|
| 存储位置 | JVM 堆内存 | PostgreSQL 数据库 |
| 持久化 | ❌ 应用重启数据丢失 | ✅ 数据库持久化 |

| 容量 | 受 JVM 内存限制 | 受磁盘大小限制 |

| 适用场景 | 本地开发/原型验证/小数据量 | 生产环境/大数据量/多服务共享 |

**Q2: 什么是 HNSW 索引？为什么选择 HNSW？**

**答**：
- **HNSW**（Hierarchical Navigable Small World）：层次可导航小世界图，一种近似最近邻（ANN）搜索算法
- **原理**：构建多层图结构，顶层稀疏（长距离跳转），底层密集（精确搜索），搜索时从顶层快速定位到大致区域再逐层细化
- **优势**：
  - 查询速度极快（O(log N)）
  - 召回率高（通常 > 95%）
  - 支持动态插入
- **代价**：索引构建时间较长，内存占用较大
- **对比 IVFFlat**：IVFFlat 构建快但查询慢，适合静态数据；HNSW 查询快，适合频繁检索的 RAG 场景

**Q3: 什么是查询重写（Query Rewrite）？为什么需要？**

**答**：
- **定义**：在 RAG 检索前，用 LLM 将用户口语化的自然语言问题改写为更适合检索的关键词查询
- **为什么需要**：
  - 用户问"我该怎么追喜欢的女生？"→ 改写为"追求心仪对象的方法 恋爱技巧"
  - Embedding 模型对口语化长句的向量表示不够精准，改写后的关键词查询检索命中率更高
- **多查询扩展**：从一个问题扩展为 3 个不同角度的查询，合并检索结果，提高召回率

**Q4: 什么是大模型幻觉（Hallucination）？如何缓解？**

**答**：
- **幻觉**：模型生成的内容与事实不符、逻辑不通或完全编造
  - **内在幻觉**：生成内容与 Prompt 中的信息矛盾
  - **外在幻觉**：生成现实中不存在的信息（编造网址、数据、人名）
- **缓解策略**：
  1. **RAG 约束**：基于检索到的真实文档回答，`allowEmptyContext(false)` + `emptyContextPromptTemplate("抱歉，我只能回答恋爱相关的问题...")` 确保无匹配文档时拒绝回答
  2. **Temperature 降低**：降低随机性（但会减少创造性）
  3. **提示词约束**：明确要求"如果不知道就说不知道"
  4. **引用来源**：要求模型标注信息来源

#### 6.2 面试题与参考答案

**Q1: 什么是 Tool Calling（Function Calling）？它的完整流程是怎样的？**

**答**：
**Tool Calling** 允许 LLM 在需要外部信息或执行操作时"调用"预定义的工具函数。

**完整流程**：
```
1. 用户发送消息："帮我搜索上海的约会地点"
2. 应用将消息 + 可用工具列表（含 JSON Schema 描述）发送给 AI
3. AI 分析后决定：需要调用 searchWeb(query="上海 约会地点")
4. AI 返回 ToolCall 对象（不执行，只指定工具名和参数）
5. 应用程序收到 ToolCall → 真正调用 Java 方法 → 获取结果
6. 应用程序将工具结果追加到对话上下文 → 再次调用 AI
7. AI 根据工具结果生成最终的自然语言回复
```

**关键点**：AI **不执行工具**，它只返回"我想调用哪个工具、传什么参数"，实际执行由应用程序完成。

**Q2: `@Tool(description = "...")` 的 description 有多重要？**

**答**：
**极其重要**。description 是 AI 决定"何时调用哪个工具"的**唯一依据**。
- 如果 description 写得不清晰，AI 可能：不该调工具时调了、该调时不调、调了错误的工具
- 最佳实践：用英文写描述（大多数模型的训练数据以英文为主），清晰说明工具功能、适用场景和参数含义
- 示例：
  ```java
  @Tool(description = "Search for information from Baidu Search Engine. " +
      "Use this when you need real-time or factual information from the internet.")
  public String searchWeb(@ToolParam(description = "Search query keyword") String query)
  ```

**Q3: 为什么工具返回值统一使用 String？**

**答**：
1. **AI 只理解文本**：模型不能直接处理 Java 对象，工具返回的 String 会被添加到对话上下文中作为 AI 下一步推理的依据
2. **简化实现**：工具内部用 try-catch 捕获所有异常，返回友好的错误描述字符串而非抛出异常
3. **Token 控制**：返回的 String 长度直接影响 Token 消耗，需要做截断处理（如 `MAX_TOOL_OUTPUT_LENGTH = 3000`）

**Q4: 工具调用有哪些安全风险？如何防范？**

**答**：
| 风险 | 说明 | 防范措施 |
|------|------|---------|
| 命令注入 | `TerminalOperationTool` 执行任意命令 | 命令白名单、禁用 `rm -rf` 等危险操作 |
| 路径穿越 | `FileOperationTool` 读写任意文件 | 限制操作目录到 `FILE_SAVE_DIR`（沙箱） |
| 信息泄露 | `WebScrapingTool` 抓取内网地址 | URL 黑白名单、禁止访问 localhost/内网 IP |
| 资源耗尽 | 下载超大文件、抓取超大网页 | 超时控制（10s）、输出截断（2000-3000 字符） |
| DoS | 恶意高频调用 SearchAPI | 频率限制、API Key 使用配额 |

#### 7.2 面试题与参考答案

**Q1: 什么是 MCP？它解决了什么问题？**

**答**：
- **MCP**（Model Context Protocol）是 Anthropic 提出的**开放标准协议**，用于标准化 AI 模型与外部工具/数据源之间的通信
- **解决的问题**：在 MCP 之前，每个 AI 应用都要为每个工具单独编写集成代码（N 个工具 = N 种集成方式）。MCP 统一了工具调用协议，类似于 USB-C 统一了设备接口
- **核心价值**：
  - **一次开发，多处使用**：MCP Server 可以被任何支持 MCP 的 Client 调用（不限于 Java）
  - **跨语言**：Server 可以用 Java、Python、Node.js 等任何语言开发
  - **工具发现**：Client 自动发现 Server 提供的工具列表，无需手动配置

**Q2: MCP 的 STDIO 和 SSE 模式有什么区别？如何选择？**

**答**：

| 维度 | STDIO | SSE |
|------|-------|-----|
| 通信方式 | 进程 stdin/stdout | HTTP（Server 独立进程） |
| 生命周期 | Client 启动时 spawn 子进程 | Server 独立运行，Client 连接 |
| 进程模型 | Client 管理子进程 | 各自独立 |
| 网络 | 无网络开销，只能本机 | 可以远程部署 |
| 多客户端 | 1:1（每个 Client 一个子进程） | 1:N（一个 Server 多 Client） |
| 调试 | stdio 被 JSON-RPC 占用，不能打印日志 | 独立进程，日志和协议分离 |

**选择**：
- 本地开发/单机部署 → STDIO（安全、零延迟）
- 远程部署/多服务共享 → SSE（网络可达、多客户端）

**Q3: 为什么 STDIO 模式下不能使用 `System.out.println`？**

**答**：
STDIO 模式下，MCP Server 通过标准输入/输出流与 Client 进行 JSON-RPC 2.0 通信：
- `stdin`：接受 Client 发来的 JSON-RPC 请求
- `stdout`：向 Client 发送 JSON-RPC 响应

如果在 Java 代码中使用 `System.out.println("日志信息")`，这些字符串会混入 JSON-RPC 响应流中，破坏协议格式，导致 Client 解析失败。因此所有日志必须通过日志框架（SLF4J/Logback）输出到文件或 stderr。

**Q4: MCP 调用的本质是什么？是 AI 服务器直接调用 MCP 服务吗？**

**答**：
**不是**。MCP 调用的本质是**工具调用的标准化**，调用链路如下：
```
用户消息 → 主应用（Spring Boot）
  → ChatClient 发送给 DashScope AI
  → AI 决策：调用高德地图 MCP 工具
  → 返回 ToolCall 给主应用
  → 主应用通过 MCP 协议向 MCP Server 发起 JSON-RPC 调用
  → MCP Server 执行查询，返回结果
  → 主应用将结果追加到对话上下文
  → 再次调用 AI
  → AI 基于结果生成自然语言回复
```

AI 服务器（DashScope）**不知道 MCP 协议的存在**，它只返回"我想调用工具 X(Y)"的指令，主应用作为中间人代为执行。

| 8.10 | **防护机制** | `MAX_TOOL_OUTPUT_LENGTH = 3000`、`MAX_CONSECUTIVE_ERRORS = 3`、`maxSteps = 20` |
| 8.11 | **SSE 流式执行** | `CompletableFuture.runAsync()` 异步、`SseEmitter` 实时推送、手动停止 |

| 8.14 | **测试策略** | Stub Agent、Mock ToolCallingManager、直接状态注入 |

#### 8.2 面试题与参考答案

**Q1: 什么是 AI Agent？它和普通 Chatbot 有什么区别？**

**答**：
- **AI Agent（智能体）** = LLM + 工具 + 自主规划 + 记忆系统
- **核心区别**：

| 维度 | 普通 Chatbot | AI Agent |
|------|-------------|----------|
| 交互模式 | 一问一答（被动） | 自主多步执行（主动） |
| 工具使用 | 无或单次调用 | 多工具组合，多轮调用 |
| 规划能力 | 无 | 自主分解复杂任务 |
| 终止决策 | 回答完成 | 主动调用 terminate 工具 |
| 状态管理 | 无状态 | 有内部状态和生命周期 |

- **类比**：Chatbot 是"客服"，回答一个问题就结束；Agent 是"助理"，给他一个复杂任务，他会自己规划步骤、调用工具、逐步完成

**Q2: 什么是 ReAct 模式？它的循环流程是怎样的？**

**答**：
ReAct = **Re**asoning + **Act**ing，是一种让 LLM 交替进行推理和行动的提示模式：

```
Cycle:
  1. Think（推理）：
     - AI 分析当前状态："用户想搜索约会地点并生成 PDF"
     - AI 决定行动："我需要先搜索网络获取地点信息"
     - 返回 ToolCall: searchWeb("上海约会地点")
     
  2. Act（行动）：
     - 执行 searchWeb 工具
     - 获取搜索结果
     
  3. Observe（观察）：
     - 工具结果自动加入对话上下文
     - 成为下一步 Think 的输入
     
  4. 循环直到 FINISHED（调用 terminate）或达到 maxSteps
```

**核心 Insight**：每次工具执行的结果自动成为下一轮推理的上下文，形成**自主推进的正反馈循环**。

**Q3: 为什么 `withInternalToolExecutionEnabled(false)` 是 Agent 与普通 ChatClient 的本质区别？**

**答**：
- **普通 ChatClient**：启用内置工具执行（`true`），Spring AI 自动管理消息上下文。一次 `call()` 中，AI 返回 ToolCall → Spring AI 自动执行工具 → 自动将结果追加到上下文 → 自动再次调用 AI → 返回最终结果。这是**黑盒**执行
- **Agent 场景**：需要在 for 循环中**多次调用 AI + 多次执行工具**，每次循环的上下文由 Agent 自己管理（`setMessageList(toolExecutionResult.conversationHistory())`）。如果启用内置执行，Spring AI 会在一次 call 中"替 Agent 完成"所有工具调用和执行，Agent 失去了对执行流程的控制
- **本质区别**：Agent 需要**白盒控制**执行流程，内置执行会破坏 Agent Loop 的规划

**Q4: 为什么 VioManus 不能作为 Spring 单例 Bean？**

**答**：
Agent 维护**内部状态**（`state`、`currentStep`、`messageList`、`stopped`），这些状态属于**单次请求**：
- 如果作为单例 Bean，用户 A 和用户 B 的请求会共享同一个 `messageList`，导致对话混乱
- 状态冲突：用户 A 的请求正在 RUNNING，用户 B 的请求来时会因为状态不是 IDLE 而抛异常
- **正确做法**：每次请求在 Controller 中 `new VioManus(...)` 创建新实例

**Q5: Agent 的防护机制有哪些？为什么需要？**

**答**：

| 防护机制 | 值 | 保护目的 |
|----------|-----|---------|
| `maxSteps` | 20 | 防止 AI 陷入无限循环（如反复调用工具无进展）持续消耗 Token |
| `MAX_TOOL_OUTPUT_LENGTH` | 3000 | 防止工具返回超长内容（如 20 万字符的 HTML）塞爆上下文窗口 |
| `MAX_CONSECUTIVE_ERRORS` | 3 | 防止 API 调用失败或网络异常导致死循环 |
| `volatile stopped` | - | 用户可随时手动停止长时间运行的任务 |
| `state` 校验 | - | 防止从非 IDLE 状态重复启动（状态保护） |

这都是**成本控制**和**用户体验**的平衡：Agent 自主运行但必须有安全阀。

#### 9.2 面试题与参考答案

**Q1: Nginx 配置 SSE 流式输出时最重要的是什么？**

**答**：
**`proxy_buffering off`** 是最关键的配置。

- **原理**：Nginx 默认会缓冲上游服务器的响应，等积累到一定大小后再发送给客户端。这对普通 HTTP 响应没问题，但对 SSE 是致命的——所有实时推送的数据会被积压在缓冲区，直到连接关闭才一次性发送
- **额外配置**：
  - `chunked_transfer_encoding off`：关闭分块传输编码
  - `proxy_read_timeout 600s`：Agent 执行可能长达几分钟，默认 60s 超时不够
  - `proxy_http_version 1.1`：支持长连接

**Q2: Vue Router 的 History 模式需要什么服务端配置？**

**答**：
Nginx 需要配置 `try_files $uri $uri/ /index.html`：
- 用户直接访问 `/love-master` → Nginx 找不到对应的静态文件 → fallback 到 `index.html` → Vue Router 接管路由 → 渲染正确页面
- 没有这个配置，刷新页面会返回 404

**Q3: Docker 多阶段构建的好处是什么？**

**答**：
前端 Dockerfile 使用 2 个阶段：
1. **Stage 1（Node 编译）**：基于 `node:20-alpine`，执行 `npm install` + `npm run build`，产出 dist 目录（编译后的静态文件）
2. **Stage 2（Nginx 托管）**：基于 `nginx:alpine`，只复制 Stage 1 产出的 dist → `/usr/share/nginx/html`，加上 `nginx.conf`

**好处**：
- 最终镜像**只包含 Nginx + 静态文件**（~20MB），不包含 Node.js、node_modules、源码
- 镜像更小 → 拉取更快 → 启动更快 → 安全面更小

**Q4: 前端如何根据环境自动切换 API 地址？**

**答**：
```javascript
const API_BASE_URL = import.meta.env.MODE === 'production'
  ? '/api'                         // 生产：Nginx 反向代理，同源请求
  : 'http://localhost:8123/api'    // 开发：CORS 跨域直连后端
```
- `import.meta.env.MODE` 是 Vite 的环境变量，`npm run dev` 时为 `development`，`npm run build` 时为 `production`
- 生产环境使用**相对路径** `/api`，因为 Nginx 将前端和后端部署在同一域名下

## 三、综合面试题（跨阶段、系统性）

### Z1: 请描述这个项目的完整架构，从用户发送消息到 AI 返回结果的全链路。

**答**：
```
1. 用户在浏览器中输入消息（Vue 3 前端）
2. 前端通过 EventSource 连接后端 SSE 端点
3. 后端 Controller 接收请求，创建 VioManus 实例
4. VioManus.runStream() 启动 Agent Loop：
   a. Think：ChatClient 将 [SystemPrompt + MessageList + nextStepPrompt + 工具列表]
      发送给 DashScope AI
   b. AI 分析任务，决定调用哪个工具（如 searchWeb）
   c. AI 返回 ToolCall（工具名 + 参数）
   d. Act：ToolCallingManager 执行 Java 工具方法
   e. 工具结果追加到 MessageList
   f. 如果调用了 terminate → state = FINISHED → 退出循环
   g. 否则回到步骤 a，继续下一轮 Think
5. 每步结果通过 SseEmitter 实时推送到前端
6. 前端 EventSource.onmessage 接收数据，更新 UI
```

### Z2: 这个项目中使用了哪些设计模式？分别在什么场景？

| 设计模式 | 应用位置 | 说明 |
|----------|---------|------|
| **模板方法** | `BaseAgent.run()` / `ReActAgent.step()` | 父类定义执行骨架，子类填充 `think()`/`act()` |
| **策略模式** | `ToolCallback[]` 可插拔工具 | 不同工具组合构造不同能力的 Agent |
| **状态模式** | `AgentState` 枚举 | IDLE→RUNNING→FINISHED/ERROR 生命周期状态转换 |
| **责任链** | `Advisor` 拦截器链 | 多个 Advisor 依次处理请求/响应 |
| **工厂方法** | `Result.success()`/`Result.error()` | 统一响应体创建 |
| **单例模式** | Spring Bean（`@Component`） | 无状态组件（LoveApp、工具类） |
| **原型模式** | VioManus 每次请求 new | 有状态 Agent 不能共享 |

### Z3: 项目中如何处理错误和异常？有哪些层次的容错？

**答**：
1. **HTTP 层**：`GlobalExceptionHandler` 捕获 11 种异常类型 + 兜底 Exception → 统一返回 `Result.error()`
2. **业务层**：`BusinessException` 携带 code + message，异常链保留根因
3. **AI 层**：`doOnError` 保护流式调用、Embedding API 欠费降级为空向量库
4. **Agent 层**：
   - `MAX_CONSECUTIVE_ERRORS = 3` → 自动终止
   - 工具内部 try-catch → 返回错误描述字符串而非抛异常
   - `volatile stopped` → 用户手动停止
5. **部署层**：`@Autowired(required = false)` + `@Lazy` 实现渐进式可用，可选组件未就绪不影响核心功能

### Z4: 如果要你优化这个项目的 Token 消耗，你会从哪些方面入手？

**答**：
1. **System Prompt 精简**：当前 `VioManus.SYSTEM_PROMPT` 较长，每次 Think 都发送，可用更简练的版本
2. **滑动窗口记忆**：`MessageWindowChatMemory.maxMessages(20)` 已经实现，可进一步减少到 10
3. **工具输出截断**：`MAX_TOOL_OUTPUT_LENGTH = 3000` 已在做，可以更激进（1000-1500）
4. **工具描述精简**：`@Tool(description)` 描述也会消耗 Token，保持简洁
5. **任务复杂度路由**：简单对话 → LoveApp（Token 少），复杂任务 → VioManus（Token 多）
6. **maxSteps 动态调整**：简单任务 5 步足够，不必固定 20
7. **模型选择**：简单任务用便宜模型（qwen-turbo），复杂推理用强模型（qwen-max）
