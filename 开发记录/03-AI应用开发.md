## 二、核心知识点

### 2.1 Prompt 工程基本概念

**Prompt 工程（Prompt Engineering）** 又叫提示词工程，即输入给 AI 的指令。提示词的质量直接影响到 AI 大模型输出的结果，是 AI 应用开发的关键技能。

**学习目标**：通过精心设计和优化输入提示来引导 AI 模型生成符合预期的高质量输出。

### 2.2 提示词分类

#### 基于角色的分类（核心）

| 类型 | 说明 | 示例 |
|------|------|------|
| 用户 Prompt（User） | 用户向 AI 提供的实际问题、指令或信息 | "帮我写一首关于春天的短诗" |
| 系统 Prompt（System） | 设置 AI 行为规则和角色定位的隐藏指令 | "你是一位经验丰富的恋爱顾问..." |
| 助手 Prompt（Assistant） | AI 模型的响应内容，在多轮对话中成为上下文 | "我是你的恋爱顾问，很高兴能帮助你..." |

> 不同的系统 Prompt 可以让同一个 AI 模型表现出完全不同的应用特性，这是构建垂直领域 AI 应用的关键。

#### 基于功能的分类（扩展）

- **指令型**：明确告诉 AI 需要执行的任务
- **对话型**：模拟自然对话，以问答形式交互
- **创意型**：引导 AI 进行创意内容生成
- **角色扮演**：让 AI 扮演特定角色或人物回答
- **少样本学习**：提供示例引导 AI 理解输出格式和风格

#### 基于复杂度的分类（扩展）

- **简单提示词**：单一指令或问题
- **复合提示词**：包含多个相关指令或步骤
- **链式提示词**：一系列连续的、相互依赖的提示词
- **模板提示词**：包含可替换变量的标准化提示词结构，如 `"你是一位专业的{领域}专家。请回答以下关于{主题}的问题：{具体问题}"`

### 2.3 Token 概念与成本优化

**Token** 是大模型处理文本的基本单位，可能是单词或标点符号。一般 Token 越多，成本越高，输出速度越慢。

**估算规则**：
- 英文文本：1 token 约等于 4 个字符或约 0.75 个英文单词
- 中文文本：一个汉字通常为 1-2 个 token
- 100 个英文单词约等于 75-150 Token，100 个中文字符约等于 100-200 Token

**成本公式**：`总成本 = (输入 token 数 x 输入单价) + (输出 token 数 x 输出单价)`

**成本优化技巧**：
1. **精简系统提示词**：移除冗余表述，保留核心指令
2. **定期清理对话历史**：请求 AI 总结之前的对话，以总结替代详细历史
3. **使用向量检索代替直接输入**：使用 RAG 技术获取相关段落，而非将整个文档作为 Prompt
4. **结构化替代自然语言**：使用表格、列表等结构化格式代替长段落描述

### 2.4 Prompt 优化技巧

#### 基础提示技巧

| 技巧 | 说明 | 核心要点 |
|------|------|----------|
| 明确指定任务和角色 | 为 AI 提供清晰的任务描述和角色定位 | 帮助模型理解背景和期望 |
| 提供详细说明和具体示例 | 提供足够的上下文信息和期望的输出格式 | 减少模型的不确定性 |
| 使用结构化格式引导思维 | 通过列表、表格等结构化格式 | 使指令更易理解，输出更有条理 |
| 明确输出格式要求 | 指定输出的格式、长度、风格等要求 | 获得更符合预期的结果 |

#### 进阶提示技巧

| 技巧 | 说明 | 适用场景 |
|------|------|----------|
| 思维链提示法（Chain-of-Thought） | 引导模型展示推理过程，逐步思考 | 提高复杂问题的准确性 |
| 少样本学习（Few-Shot Learning） | 提供几个输入-输出对示例 | 帮助模型理解任务模式 |
| 分步骤指导（Step-by-Step） | 将复杂任务分解为可管理的步骤 | 确保模型完成每个关键环节 |
| 自我评估和修正 | 让模型评估自己的输出并改进 | 提高准确性和质量 |
| 知识检索和引用 | 引导模型检索相关信息并明确引用来源 | 提高可靠性 |
| 多视角分析 | 从不同角度、立场或专业视角分析 | 提供全面见解 |
| 多模态思维 | 结合不同表达形式进行思考 | 复杂系统设计 |

**总结**：任务越复杂，就越要给 Prompt 补充更多细节。把 AI 当成人类，模糊的问题得到模糊的答案。

#### 提示词调试与优化

1. **迭代式提示优化**：逐步修改和完善提示词，从笼统到具体
2. **边界测试**：通过极限情况测试模型的能力边界
3. **提示词模板化**：创建结构化模板，便于一致性提示
4. **错误分析与修正**：系统性分析模型回答中的错误，针对性优化

### 步骤 3：理解 Spring AI 核心特性

#### 3.1 ChatClient

ChatClient 支持更复杂灵活的链式调用（Fluent API），与 ChatModel 的区别：

```java
// ChatModel 方式
ChatResponse response = chatModel.call(new Prompt("你好"));

// ChatClient 方式（推荐）
ChatClient chatClient = ChatClient.builder(chatModel)
    .defaultSystem("你是恋爱顾问")
    .build();

String response = chatClient.prompt().user("你好").call().content();
```

**构建 ChatClient**：

```java
@Service
public class ChatService {
    private final ChatClient chatClient;

    public ChatService(ChatClient.Builder builder) {
        this.chatClient = builder
            .defaultSystem("你是恋爱顾问")
            .build();
    }
}
```

**ChatClient 支持的响应格式**：

```java
// 返回 ChatResponse 对象
ChatResponse chatResponse = chatClient.prompt()
    .user("Tell me a joke")

// 返回实体对象
record ActorFilms(String actor, List<String> movies) {}
ActorFilms actorFilms = chatClient.prompt()
```

#### 3.2 Advisors（顾问 / 拦截器机制）

Spring AI 使用 Advisors 机制来增强 AI 的能力，可以理解为一系列可插拔的拦截器：

- **前置增强**：调用 AI 前改写 Prompt、检查安全性
- **后置增强**：调用 AI 后记录日志、处理返回结果

**执行流程**：

1. Spring AI 从用户的 Prompt 创建一个 AdvisedRequest，同时创建空的 AdvisorContext
2. 链中的每个 Advisor 处理该请求，可修改或阻止请求
3. 最终的 Advisor 将请求发送给 ChatModel
4. 聊天模型的响应通过 Advisor 链传回，转换为 AdvisedResponse
5. 每个 Advisor 可处理或修改响应
6. 最终的 AdvisedResponse 返回给客户端

**执行顺序**：通过 `getOrder()` 方法决定，值越低越优先执行，而非代码编写顺序决定。

**Advisor 分为两种模式**：
- **流式（Streaming）**：用于流式请求和响应
- **非流式（Non-Streaming）**：用于同步请求和响应

#### 3.3 Chat Memory Advisor

实现对话记忆功能的 Advisor：

| 实现类 | 说明 |
|--------|------|
| MessageChatMemoryAdvisor | 将历史对话作为消息集合添加到提示词中，保留原始对话结构 |
| PromptChatMemoryAdvisor | 将对话历史添加到系统文本中，可能失去消息边界 |
| VectorStoreChatMemoryAdvisor | 使用向量数据库存储和检索历史对话 |

> **推荐使用 MessageChatMemoryAdvisor**：更符合现代 LLM 的对话模型设计，能更好地保持上下文连贯性。

#### 3.4 Chat Memory

Chat Memory 负责历史对话的存储，Spring AI 内置了几种实现：

| 实现类 | 存储方式 |
|--------|----------|
| InMemoryChatMemory | 内存存储 |
| CassandraChatMemory | Cassandra 持久化 |
| Neo4jChatMemory | Neo4j 持久化 |
| JdbcChatMemory | JDBC 持久化 |

可自定义实现 ChatMemory 接口来扩展存储方式。

### 步骤 4：多轮对话 AI 应用开发

在 `app` 包下创建 `LoveApp.java`：

```java
@Component
@Slf4j
public class LoveApp {

    private final ChatClient chatClient;

    private static final String SYSTEM_PROMPT = "扮演深耕恋爱心理领域的专家。开场向用户表明身份，告知用户可倾诉恋爱难题。" +
            "围绕单身、恋爱、已婚三种状态提问：单身状态询问社交圈拓展及追求心仪对象的困扰；" +
            "恋爱状态询问沟通、习惯差异引发的矛盾；已婚状态询问家庭责任与亲属关系处理的问题。" +
            "引导用户详述事情经过、对方反应及自身想法，以便给出专属解决方案。";

    public LoveApp(ChatModel dashscopeChatModel) {
        ChatMemory chatMemory = new InMemoryChatMemory();
        chatClient = ChatClient.builder(dashscopeChatModel)
                .defaultSystem(SYSTEM_PROMPT)
                .defaultAdvisors(
                        new MessageChatMemoryAdvisor(chatMemory)
                )
                .build();
    }

    public String doChat(String message, String chatId) {
        ChatResponse response = chatClient
                .prompt()
                .user(message)
                .advisors(spec -> spec.param(CHAT_MEMORY_CONVERSATION_ID_KEY, chatId)
                        .param(CHAT_MEMORY_RETRIEVE_SIZE_KEY, 10))
                .call()
                .chatResponse();
        String content = response.getResult().getOutput().getText();
        log.info("content: {}", content);
        return content;
    }
}
```

### 步骤 5：自定义 Advisor

#### 自定义日志 Advisor

```java
@Slf4j
public class MyLoggerAdvisor implements CallAroundAdvisor, StreamAroundAdvisor {

    @Override
    public String getName() {
        return this.getClass().getSimpleName();
    }

    @Override
    public int getOrder() {
        return 0;
    }

    private AdvisedRequest before(AdvisedRequest request) {
        log.info("AI Request: {}", request.userText());
        return request;
    }

    private void observeAfter(AdvisedResponse advisedResponse) {
        log.info("AI Response: {}", advisedResponse.response().getResult().getOutput().getText());
    }

    public AdvisedResponse aroundCall(AdvisedRequest advisedRequest, CallAroundAdvisorChain chain) {
        advisedRequest = this.before(advisedRequest);
        AdvisedResponse advisedResponse = chain.nextAroundCall(advisedRequest);
        this.observeAfter(advisedResponse);
        return advisedResponse;
    }

    public Flux<AdvisedResponse> aroundStream(AdvisedRequest advisedRequest, StreamAroundAdvisorChain chain) {
        advisedRequest = this.before(advisedRequest);
        Flux<AdvisedResponse> advisedResponses = chain.nextAroundStream(advisedRequest);
        return (new MessageAggregator()).aggregateAdvisedResponse(advisedResponses, this::observeAfter);
    }
}
```

#### 自定义 Re-Reading Advisor（Re2 技术）

通过让模型重新阅读问题来提高推理能力：

```java
public class ReReadingAdvisor implements CallAroundAdvisor, StreamAroundAdvisor {

    private AdvisedRequest before(AdvisedRequest advisedRequest) {
        Map<String, Object> advisedUserParams = new HashMap<>(advisedRequest.userParams());
        advisedUserParams.put("re2_input_query", advisedRequest.userText());

        return AdvisedRequest.from(advisedRequest)
                .userText("""
                        {re2_input_query}
                        Read the question again: {re2_input_query}
                        """)
                .userParams(advisedUserParams)
                .build();
    }

    @Override
    public AdvisedResponse aroundCall(AdvisedRequest advisedRequest, CallAroundAdvisorChain chain) {
        return chain.nextAroundCall(this.before(advisedRequest));
    }

    @Override
    public Flux<AdvisedResponse> aroundStream(AdvisedRequest advisedRequest, StreamAroundAdvisorChain chain) {
        return chain.nextAroundStream(this.before(advisedRequest));
    }

    @Override
    public int getOrder() {
        return 0;
    }

    @Override
    public String getName() {
        return this.getClass().getSimpleName();
    }
}
```

> 注意：Re2 技术虽然可提高推理能力，但成本会加倍，面向 C 端开放的应用不建议使用。

**在 LoveApp 中应用自定义 Advisor**：

```java
chatClient = ChatClient.builder(dashscopeChatModel)
    .defaultSystem(SYSTEM_PROMPT)
    .defaultAdvisors(
        new MessageChatMemoryAdvisor(chatMemory),
        new MyLoggerAdvisor(),
        new ReReadingAdvisor()
    )
    .build();
```

**Advisor 最佳实践**：
1. 保持单一职责：每个 Advisor 专注于一项特定任务
2. 注意执行顺序：合理设置 `getOrder()` 值
3. 同时支持流式和非流式：实现两种接口以提高灵活性
4. 高效处理请求：避免在 Advisor 中执行耗时操作
5. 测试边界情况：确保优雅处理异常和边界情况
6. 可使用 `adviseContext` 在 Advisor 链中共享状态

### 步骤 6：结构化输出 - 恋爱报告功能

#### 结构化输出原理

**StructuredOutputConverter** 在大模型调用前后都发挥作用：
- **调用前**：在提示词后面附加格式指令，引导模型生成符合指定格式的响应
- **调用后**：将模型的文本输出转换为结构化类型实例

**核心接口设计**：

```java
public interface StructuredOutputConverter<T> extends Converter<String, T>, FormatProvider {
}
```

**Spring AI 提供的转换器**：

| 转换器 | 用途 |
|--------|------|
| BeanOutputConverter | 将输出转换为 Java Bean 对象 |
| MapOutputConverter | 将输出转换为 Map 结构 |
| ListOutputConverter | 将输出转换为 List 结构 |

**使用示例**：

```java
// Bean 转换
record ActorsFilms(String actor, List<String> movies) {}
ActorsFilms actorsFilms = ChatClient.create(chatModel).prompt()
    .user("Generate 5 movies for Tom Hanks.")
    .call()
    .entity(ActorsFilms.class);

// 复杂泛型转换
List<ActorsFilms> actorsFilms = ChatClient.create(chatModel).prompt()
    .user("Generate the filmography of 5 movies for Tom Hanks and Bill Murray.")
    .call()
    .entity(new ParameterizedTypeReference<List<ActorsFilms>>() {});

// Map 转换
Map<String, Object> result = ChatClient.create(chatModel).prompt()
    .user(u -> u.text("Provide me a List of {subject}")
        .param("subject", "an array of numbers from 1 to 9"))
    .call()
    .entity(new ParameterizedTypeReference<Map<String, Object>>() {});
```

### 步骤 7：对话记忆持久化

#### 自定义文件持久化 ChatMemory

Spring AI 解耦了"存储"和"记忆算法"，可通过自定义 ChatMemory 改变存储位置。

**难点**：Message 是接口，有多种子类实现，各子类字段不同，没有无参构造函数，未实现 Serializable 接口。

**解决方案**：使用 Kryo 高性能序列化库。

**引入依赖**（pom.xml）：

```xml
<dependency>
    <groupId>com.esotericsoftware</groupId>
    <artifactId>kryo</artifactId>
    <version>5.6.2</version>
</dependency>
```

**核心实现逻辑**：

```java
public class FileBasedChatMemory implements ChatMemory {

    private final String BASE_DIR;
    private static final Kryo kryo = new Kryo();

    static {
        kryo.setRegistrationRequired(false);
        kryo.setInstantiatorStrategy(new StdInstantiatorStrategy());
    }

    public FileBasedChatMemory(String dir) {
        this.BASE_DIR = dir;
        File baseDir = new File(dir);
        if (!baseDir.exists()) {
            baseDir.mkdirs();
        }
    }

    @Override
    public void add(String conversationId, List<Message> messages) {
        List<Message> conversationMessages = getOrCreateConversation(conversationId);
        conversationMessages.addAll(messages);
        saveConversation(conversationId, conversationMessages);
    }

    @Override
    public List<Message> get(String conversationId, int lastN) {
        List<Message> allMessages = getOrCreateConversation(conversationId);
        return allMessages.stream()
                .skip(Math.max(0, allMessages.size() - lastN))
                .toList();
    }

    @Override
    public void clear(String conversationId) {
        File file = getConversationFile(conversationId);
        if (file.exists()) {
            file.delete();
        }
    }

    private List<Message> getOrCreateConversation(String conversationId) {
        File file = getConversationFile(conversationId);
        List<Message> messages = new ArrayList<>();
        if (file.exists()) {
            try (Input input = new Input(new FileInputStream(file))) {
                messages = kryo.readObject(input, ArrayList.class);
            } catch (IOException e) {
                e.printStackTrace();
            }
        }
        return messages;
    }

    private void saveConversation(String conversationId, List<Message> messages) {
        File file = getConversationFile(conversationId);
        try (Output output = new Output(new FileOutputStream(file))) {
            kryo.writeObject(output, messages);
        } catch (IOException e) {
            e.printStackTrace();
        }
    }

    private File getConversationFile(String conversationId) {
        return new File(BASE_DIR, conversationId + ".kryo");
    }
}
```

**在 LoveApp 中使用**：

### 步骤 8：PromptTemplate 模板

**PromptTemplate** 是 Spring AI 中用于构建和管理提示词的核心组件，支持变量替换：

```java
String template = "你好，{name}。今天是{day}，天气{weather}。";

PromptTemplate promptTemplate = new PromptTemplate(template);

Map<String, Object> variables = new HashMap<>();
variables.put("name", "鱼皮");
variables.put("day", "星期一");
variables.put("weather", "晴朗");

String prompt = promptTemplate.render(variables);
```

**专用模板类**：

| 模板类 | 用途 |
|--------|------|
| SystemPromptTemplate | 用于系统消息，设置 AI 的行为和背景 |
| AssistantPromptTemplate | 用于助手消息，设置 AI 回复的结构 |

**从文件加载模板**：

```java
@Value("classpath:/prompts/system-message.st")
private Resource systemResource;

SystemPromptTemplate systemPromptTemplate = new SystemPromptTemplate(systemResource);
```

> 推荐使用文件管理 Prompt 模板：将复杂提示词放在单独文件中管理，无需修改代码即可调整，为不同场景准备多套模板。

### 步骤 9：多模态开发

**原生多模态大模型**：在架构设计和预训练阶段就直接整合多种数据类型，如 GPT-4o、Gemini 1.5、Claude3。

## 四、技术要点总结

1. **Prompt 是 AI 应用的灵魂**：系统提示词直接决定 AI 的行为模式，不同的提示词让同一模型表现出完全不同的应用特性。
2. **ChatClient 优于 ChatModel**：ChatClient 支持链式调用、默认系统提示词、Advisor 拦截器、结构化输出等高级功能。
3. **Advisor 即拦截器**：采用责任链模式，支持前置/后置增强，可自定义实现日志、鉴权、Prompt 改写等功能。
4. **MessageChatMemoryAdvisor 是首选**：保留原始对话结构，更符合现代 LLM 的对话模型设计。
5. **结构化输出原理**：调用前附加 JSON Schema 格式指令，调用后将 JSON 文本转换为 Java 对象。
6. **Kryo 解决序列化难题**：Message 接口子类多、字段不统一、无 Serializable 实现，Kryo 是合适的序列化方案。
7. **PromptTemplate 实现前后端分离**：将提示词从代码中抽离到文件，便于版本管理和 A/B 测试。
8. **MVP 策略**：先实现核心对话功能，再逐步扩展知识库、工具调用、MCP 等高级能力。
9. **Token 成本意识**：系统提示词、用户提示词和 AI 输出都消耗成本，需持续优化。
10. **多模态是大趋势**：原生多模态大模型越来越普及，多模态开发会越来越简单。
